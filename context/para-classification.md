# PARA 분류 기준

source(외부 URL 자료)를 4개 destination 중 하나로 분류하는 기준이다. 분류기 AI(②)는 이 기준으로 **분류만** 한다 — 연결 추천이나 문서 초안은 만들지 않는다.

> 이 문서는 정의·규칙만 담는다. **애매할 때의 판단 규율(confidence·reason 작성법)은 실행 프롬프트(`classification_gate`)가, 출력 필드 계약은 `output_schema`가 소유**한다.

## Destination 4종

### `project` — 지금 실제로 만들거나 진행할 일

- 이 자료가 **진행 중인 제품/작업에 직접 반영될 구체적 액션**을 담고 있을 때.
- 대표 입력: 제품 기능 아이디어, MVP 작업 후보, 스펙 변경 제안, **기업 AX 전환 요구사항 docx**(회사가 준 요구사항 묶음).
- 승인 후: **회사 프로젝트로 팬아웃 — `projects/{corp}/`의 원본요약(baseline) + 기능정의서(spec) N장**으로 문서화된다(AXKG-SPEC-014).
- 판단 힌트: "이걸 보고 우리 제품에 무엇을 바꾸거나 만들 것인가"에 구체적 답이 나오면 project. **회사가 준 요구사항 묶음(요구 여러 개가 한 docx에)이면 project**다.
- **project 하위 분기(요구사항 vs 회사 context, AXKG-DEC-009/WORK-013)**: project 안에서도 업로드가 **요구사항**(기능으로 쪼갤 대상 → baseline+spec 팬아웃)인지 **회사 배경지식 context**(조직도·업무/휴가 플로우 등 → `projects/{corp}/context/` 단일 문서, 팬아웃 없음)인지 갈린다. **intake 메모의 성격 힌트를 우선**한다(예: "SC 회사 정보야"/"조직도" → context). 힌트가 없으면 안전 폴백은 **요구사항**(팬아웃 누락<노이즈). 회사 배경지식을 기능정의서로 쪼개지 마라.

### `area` — 계속 관리할 책임/관심 영역

- 특정 프로젝트에 종속되지 않지만 **지속적으로 쌓아갈 주제/역량**에 해당할 때.
- 대표 입력: AX 전략, AI 전환 역량, 조직 지식관리, Agent Experience 연구 주제.
- 승인 후: permanent note(영구 개념 노트) 후보로 문서화된다.
- 판단 힌트: 끝나는 시점이 없고, 개념·원칙·방법론으로 축적되는 성격이면 area.

### `resource` — 나중에 참고할 외부 자료

- 당장 액션은 없지만 **참고 가치가 있는 외부 자료**일 때. 가장 흔한 destination이다.
- 대표 입력: 기사, 유튜브, 논문, 도구 링크, 사례 분석.
- 승인 후: reference note 후보로 문서화된다.
- 판단 힌트: "언젠가 찾아볼 자료"면 resource. `resource`는 PARA 분류 라벨이고 `reference note`는 그 산출 노트 타입이다 — 같은 대상의 두 이름.

### `archive` — 지금 쓰지 않을 자료

- 보존은 하되 **문서화·연결을 하지 않을** 자료일 때.
- 대표 입력: 중복 링크, 오래된 자료, 품질 낮은 자료, 현재 범위(AI Transformation) 밖 자료.
- 승인 후: archive 처리로 종료된다. 문서화 게이트로 넘어가지 않는다.

## 경계 판단 가이드

destination은 인접한 것끼리 헷갈린다. 각 경계마다 **판별 질문**을 던지고, 답이 "예"인 쪽으로 민다.

- **project ↔ area** (마감·목표의 유무):
  - 판별 질문: "이 자료가 **끝나는 시점과 완료 기준이 있는 구체적 산출물**에 붙는가?" → 예면 project.
  - project는 마감/목표가 있는 진행 건이고, area는 마감 없이 계속 관리하는 책임·관심 영역이다. "언제 끝나는가?"에 답할 수 있으면 project, "계속 쌓는 주제"면 area.
- **project ↔ resource** (직접 액션의 유무):
  - 판별 질문: "이걸 보고 **우리 제품에 무엇을 바꾸거나 만들지 구체적 답**이 나오는가?" → 예면 project, "언젠가 유용" 수준이면 resource.
  - 자료 형식이 외부 참고물(기사·영상)이어도 내용이 구체적 제품 변경을 제안하면 project다. 형식이 아니라 **액션 가능성**으로 가른다.
- **area ↔ resource** (개념으로 소화할 가치 vs 원문 참조로 충분):
  - 판별 질문: "이걸 **개념·원칙으로 소화해 영구 노트에 녹일 가치**가 있는가, 아니면 원문 링크를 다시 찾아보는 것으로 충분한가?" → 전자면 area, 후자면 resource.
  - area는 축적되는 개념이고 resource는 참조용 원문이다. 애매하면 resource가 안전하다 — 문서화 후 파생지식으로 개념을 area로 승격·보충할 수 있다.
- **resource ↔ archive** (현재 참고가치 vs 비활성 보관):
  - 판별 질문: "지금 범위(AI Transformation)에서 **언젠가 다시 찾아볼 참고가치**가 살아 있는가?" → 예면 resource, 죽어 있으면(중복·구식·저품질·범위 밖) archive.
  - archive는 문서화·연결을 **차단**하므로 되돌리기 비싸다. 참고가치가 조금이라도 남아 있으면 resource로 둔다.

## 보수적 판정 규칙

- **archive는 가장 보수적으로**: 확신이 없으면 archive가 아니라 resource로 분류한다. archive는 문서화 파이프라인을 끊는 종착이라, 잘못된 archive가 잘못된 resource보다 손실이 크다.
- **project는 함부로 올리지 않는다**: 액션이 진짜 구체적일 때만 project. 애매하면 resource(또는 개념성이면 area)로 두고, 문서화 이후 baseline 파생으로 project화하는 경로가 열려 있다.

## 하지 말 것

- 연결 후보(`[[ ]]`, up:)를 만들지 않는다 — 연결은 문서화 게이트(③)의 초안에서 발현된다.
- destination을 복수로 제안하지 않는다 — 하나만 고르고 이유를 명확히 한다.
- 요약을 다시 쓰지 않는다 — 입력으로 받은 요약을 근거로 분류만 한다.
