# GoF 디자인 패턴

## 정의

4명의 저자(Gang of Four)가 제시한 23개의 재사용 가능한 객체지향 소프트웨어 설계 골격 구조; 특정 문맥의 공통 문제에 대한 정형화된 해결 틀로, [[생성-패턴]]·[[구조-패턴]]·[[행동-패턴]] 3범주로 분류된다.

## 맥락

디자인 패턴은 완전한 정답 알고리즘이 아니라 상황에 맞게 유연하게 수정해 사용하는 틀이다. 현재 수천 개의 디자인 패턴이 존재하나 GoF 23개를 먼저 파악하는 것이 우선이다.

세 범주 개요:

- **[[생성-패턴]]**: 객체 생성과 관련. 5종(추상 팩토리·빌더·팩토리 메소드·프로토타입·[[싱글톤-패턴]]).
- **[[구조-패턴]]**: 클래스·인터페이스·객체 조합으로 더 큰 구조를 만듦. 기존 코드 최소 변경으로 기능 추가.
- **[[행동-패턴]]**: 객체 간 알고리즘·책임 분배 정의. 3범주 중 패턴 수가 가장 많음.

장점: 개발자 간 소통 원활화, 코드 구조 파악 용이, 재사용을 통한 개발 시간 단축, 설계 변경 유연성.
단점: 언어 특성에 맞게 수정 반영 필요, 초기 투자 비용 증가.

실용적 접근: 모든 패턴 암기 불필요 — 어떤 패턴이 있는지 파악하고, 설계 단계에서 필요한 것을 검색해 적용.

Christopher Alexander의 *A Pattern Language*(건축·도시계획 패턴 카탈로그)가 이 개념의 선례이며, Sam Bhagwat(Mastra.ai)는 에이전틱 AI 패턴도 이와 같은 공통 어휘·용어집이 필요하다고 주장한다([[agents-vs-workflows-why-not-both-sam-bhagwat-mastra]] 참조).
