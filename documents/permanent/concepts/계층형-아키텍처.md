# 계층형 아키텍처(Layered Architecture)

## 정의

표현(Controller)·비즈니스(Service)·영속(Repository) 세 계층으로 코드를 분리하고, 의존 방향을 위→아래 단방향으로 고정해 변경과 테스트가 각 계층 안에 격리되도록 하는 백엔드 설계 원칙이다.

## 맥락

관심사 분리(Separation of Concerns) 원칙의 구체적 적용 형태다. 각 계층은 바뀌는 이유가 하나여야 한다는 단일 책임에 근거하며, Spring 백엔드에서 Controller·Service·Repository라는 이름으로 관례화되어 있다.

**계층별 역할과 금지 사항**:

| 계층 | 책임 | 금지 |
|------|------|------|
| Controller | 요청·응답 변환, 입력 검증(`@Valid`) | 비즈니스 규칙 판단 |
| Service | 비즈니스 규칙, 트랜잭션 경계(`@Transactional`) | HTTP·SQL 직접 처리 |
| Repository | 데이터 저장·조회 추상화 | 비즈니스 판단 |

**의존성 방향 3원칙**: (1) 단방향 — 위 계층이 아래 계층에만 의존, 역방향 금지; (2) 계층 건너뛰기 금지; (3) 추상화 의존 — 구체 구현이 아닌 인터페이스에 의존.

**DTO 경계**: 계층 간 데이터는 DTO로 변환해 엔티티를 외부에 노출하지 않는다. 엔티티 직접 노출은 내부 구조 노출과 민감 필드 유출로 이어진다.

**코드 배치 판단 기준**: "이 코드가 바뀌는 이유가 무엇인가?" — 화면 형식(Controller), 정책(Service), 저장 방식(Repository).

**실용적 분리 기준**: 모든 CRUD에 기계적으로 적용하면 보일러플레이트만 늘어난다. 변경과 테스트가 필요할 만큼 복잡도가 생길 때 분리한다.

**흔한 안티패턴**: 뚱뚱한 컨트롤러(비즈니스 로직 혼재), 빈약한 서비스(단순 호출 위임만), 엔티티 노출, 계층 침투(Service에 HTTP 객체), 양방향 의존 — 공통 원인은 한 계층이 자기 책임을 넘어서는 것이다.
