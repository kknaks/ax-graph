# 루프 엔지니어링(Loop Engineering)

## 정의

개발자가 에이전트를 직접 프롬프트하는 대신, 에이전트를 자동으로 구동하는 루프 시스템을 설계하는 AI 에이전트 운영 패러다임이다.

## 맥락

2026년 6월 Anthropic Claude Code 책임자가 "나는 더 이상 Claude에 직접 프롬프트하지 않는다. 루프가 대신 한다. 내 일은 루프를 작성하는 것이다"라고 밝히며 공식화됐다. 기존 방식(프롬프트 입력 → 결과 확인 → 다음 프롬프트)에서는 개발자 자신이 루프 역할을 했다면, 루프 엔지니어링에서는 개발자가 한 단계 올라가 에이전트가 달리는 트랙(시스템)을 설계한다.

AI 작업 패러다임 연대기: 2024년 운영자(프롬프트 작성) → 2025년 관리자(여러 에이전트 동시 운영) → 2026년 시스템 설계자(에이전트를 구동하는 시스템 설계).

루프의 4단계 반복 구조는 Discover(작업 발견) → Plan(계획) → Execute(실행) → Verify(검증) → 조건 충족 시 종료, 아니면 반복이다. 루프는 자동화·워크트리·스킬·커넥터·서브에이전트의 5가지 핵심 요소와 공유 메모리로 구성된다. 핵심 구조는 제너레이터(생성) + [[버파이어]](검증)의 쌍이며, [[버파이어]]가 루프 품질의 진짜 병목이다.

닫힌 루프(성공 조건 사전 고정, 예측 가능·저비용)와 열린 루프(목표와 탐색 여지만 부여, 창의적 탐색 가능·토큰 소모)로 나뉜다. 원칙은 작업을 수행하는 가장 덜 자율적인 도구를 선택하는 것이다. 스택 구조상 루프 엔지니어링은 단일 에이전트 실행에 필요한 모든 것(하네스)을 타이머 위에서 구동하는 외부 시스템으로, [[에이전틱-ai]] 구현 스택의 한 층 위에 위치한다.

루프의 5요소 중 **스킬(Skills)** 계층은 Microsoft Research의 [[스킬-문서|SkillOpt]](arXiv:2605.23904, 2026-05-22)로 자동 최적화가 가능해졌다. SkillOpt는 epoch·mini-batch·학습률·검증 게이트 등 딥러닝 훈련 규율을 텍스트 공간에 이식해 스킬 문서를 학습 가능한 상태로 다룬다 — 사람이 손으로 다듬는 정적 자산에서 루프가 자체 개선하는 누적 자본재로 위상이 바뀐다. 학습된 스킬은 모델·실행환경 간 이식 가능하므로(Cross-harness +31.8%), 루프를 새 하네스로 이전할 때 스킬 자산을 재사용할 수 있다.
