# 샤딩(Sharding)

## 정의

데이터를 물리적으로 다른 DB 인스턴스(샤드)로 분산·저장하는 수평 확장(스케일아웃) 전략. 샤드 키 기반 라우팅 로직을 애플리케이션 또는 미들 레이어에서 직접 구현해야 한다.

## 맥락

"옆 동네에 새 집 짓기" — [[파티셔닝]](단일 인스턴스 내 논리 분리)이 '한 지붕 아래 새 방'이라면, 샤딩은 '아예 새로운 집으로 이사'다. 단일 DB 서버의 물리적 한계(CPU·메모리·디스크)를 스케일업으로 더 이상 극복할 수 없을 때 검토하는 **최후의 수단**에 가까운 전략이다.

파티셔닝과의 결정적 차이점:

| 항목 | 파티셔닝 | 샤딩 |
|---|---|---|
| 물리적 위치 | 단일 DB 인스턴스 내부 | 다수의 별도 DB 인스턴스 |
| 확장 방향 | 스케일업 | 스케일아웃 |
| 관리 주체 | DB 엔진 자동 처리 | 애플리케이션 직접 구현 |
| SQL 변경 | 불필요 | 샤드 키 설계 필수 |

실무 적합 사례:
- **대규모 트래픽 + 대용량 데이터**: 단일 DB 한계를 넘어설 때
- **글로벌 서비스**: 지역별 데이터 물리 분산으로 데이터 로컬리티 확보

핵심 주의사항:
- **조인 연산이 사실상 불가능**해지므로 도입 전 데이터 모델링을 철저히 설계해야 한다.
- 단일 DB 한계를 명확히 확인한 후 도입을 검토한다(조기 도입은 불필요한 복잡성 유발).
