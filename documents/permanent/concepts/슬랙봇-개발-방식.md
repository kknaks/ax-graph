# 슬랙봇 개발 방식

## 정의

슬랙봇 서버를 구현하는 세 가지 접근: 알림 전용의 Incoming Webhook, 슬랙 소통 최적화 서버 프레임워크인 Bolt, 서버 선택 자유도를 제공하는 Slack SDK.

## 맥락

세 방식은 구현 범위와 서버 자유도 기준으로 구분한다.

**Incoming Webhook**: 앱 설정에서 Webhook URL을 생성하고 해당 URL을 호출하면 슬랙 메시지가 전송된다. 코드가 거의 없어 외부 서비스에서 URL만 등록하면 알림 발송이 가능하다. 단, 슬랙 메시지 전송 외 다른 동작(이벤트 수신, 명령 처리)은 불가능하다. 알림 전용으로 충분할 때 선택한다.

**Bolt 프레임워크**: 슬랙 소통에 최적화된 서버 프레임워크로 JavaScript/TypeScript·Python·Java를 지원한다. Bot Token을 `App` 객체에 연결하면 서버와 슬랙 앱이 연결되고, 슬랙 내 채팅·채널 입퇴장·아카이브 등 다양한 사용자 동작을 이벤트 리스너로 처리할 수 있다. 서버 프레임워크가 내장돼 구현 편의성이 높으나, 서버 환경 커스터마이징 자유도는 낮다.

```js
const app = new App({ token: process.env.SLACK_BOT_TOKEN });
app.message('hello', async ({ message, say }) => { await say('안녕하세요!'); });
await app.start();
```

**Slack SDK**: 슬랙 소통 기능을 제공하는 패키지(라이브러리) 단위로, Node.js·Python·Java를 지원한다. 서버 프레임워크는 별도로 선택한다(NestJS, Spring, Django 등). 핵심 패키지인 Web API는 130개 이상의 메소드를 제공한다. 에어프레미아 발표자는 서버 환경 커스터마이징 자유도를 이유로 SDK + NestJS 조합을 선택했다.

```ts
const client = new WebClient(process.env.SLACK_BOT_TOKEN);
await client.chat.postMessage({ channel, text });
```

| | Bolt | SDK |
|---|---|---|
| 서버 프레임워크 | 내장 | 직접 선택 |
| 슬랙 소통 메소드 | 내장 | 별도 패키지 |
| 자유도 | 낮음 | 높음 |
| 구현 편의성 | 높음 | 상대적으로 낮음 |

**소켓 모드(Socket Mode)**: 기업 내부망처럼 슬랙이 서버로 직접 HTTP 호출을 할 수 없는 환경에서 사용하는 설정이다. Socket Mode를 ON으로 켜고 App-Level Token을 별도 생성하면 `SocketModeClient`로 슬랙 앱과 서버를 직접 연결해 외부 URL 노출 없이 이벤트를 수신할 수 있다. Bolt와 SDK 모두 소켓 모드를 지원한다.

**inbound 엔드포인트 직접 구현 시 raw body 제약**: Bolt·SDK 모두 inbound HTTP 엔드포인트를 직접 노출할 때는 [[slack-서명검증]]을 raw body(파싱 전 bytes) 기준으로 수행해야 한다. 미들웨어나 프레임워크가 body를 form으로 파싱한 뒤 재직렬화하면 서명이 깨진다 — raw body를 미들웨어 단계에서 별도 보존해야 한다.
