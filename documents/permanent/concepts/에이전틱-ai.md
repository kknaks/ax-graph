# 에이전틱 AI(Agentic AI)

## 정의

목적성을 가진 특화 AI 에이전트가 단독 또는 멀티 에이전트 협업 방식으로 복합적 문제를 자율적으로 해결하는 AI 패러다임이다.

## 맥락

2025년 글로벌 AX 4대 핵심 트렌드 중 하나로, 기업의 과반수가 12개월 내 AI 에이전트를 주요 전략 축으로 채택할 예정이다. 단일 모델이 수동 지시를 받아 처리하는 방식과 달리, 에이전틱 AI는 목표를 부여받으면 스스로 계획·실행·검증을 반복하며 복합 업무를 처리한다. 멀티 에이전트 구성에서는 특화 에이전트들이 협업하여 단일 에이전트가 처리하기 어려운 도메인 문제를 병렬·순차적으로 해결한다. AX 맥락에서는 에이전틱 AI가 업무 자동화의 수준을 규칙 기반 자동화에서 목적 기반 자율 실행으로 격상시키는 핵심 동인으로 위치한다.

실제 적용 사례로 코르카 문라이트 코드베이스 품질 개선 워크플로우가 있다: 셸 스크립트(tidy-target-commit)로 git log에서 타이딩 대상 커밋을 필터링하고, 병렬 서브 에이전트가 각 커밋을 동시에 분석해 타이딩 제안 목록을 생성하며, 오케스트레이션 에이전트가 제안을 취합한 뒤 사람(개발자)이 최종 검토·승인한다. 이 설계의 핵심 원칙은 LLM이 할 수 있는 것과 스크립트로 할 수 있는 것을 분리해 토큰 비용과 속도를 최적화하는 것이다 — LLM은 판단·요약이 필요한 분석에만 쓰고, 기계적 필터링은 스크립트가 담당한다.

구조적 관점에서 개별 에이전트는 **턴 기반 루프**로 동작한다: 사용자(또는 시스템)가 한 턴 → 에이전트가 한 턴(툴 콜 포함) → 반복. 대화 스레드·메모리 등은 이 메시지 누적 과정에서 발생하는 emergent property다(Sam Bhagwat, Mastra.ai). 에이전틱 시스템 설계의 핵심 트레이드오프는 **자율성(Power) vs 제어(Control)**다 — 실용적 접근은 자율성으로 시작하고, 신뢰성이 낮은 지점을 식별해 구조(워크플로우 스텝 분해, 제어 로직 추가)를 삽입하는 것이다. 에이전트와 워크플로우는 상호 배타적이지 않고 [[agent-workflow-조합]] 패턴으로 중첩·조합할 수 있다.

[[루프-엔지니어링]] 패러다임(2026)은 에이전틱 AI 운영의 다음 단계를 정의한다. 개발자가 에이전트를 직접 프롬프트하는 대신 에이전트를 구동하는 루프 시스템을 설계하는 방식으로, Anthropic Claude Code 책임자가 공식화했다. AI 작업 패러다임 연대기: 2024년 운영자(프롬프트 작성) → 2025년 관리자(에이전트 동시 운영) → 2026년 시스템 설계자(에이전트를 구동하는 시스템 설계). 루프의 핵심 구조는 제너레이터(생성) + [[버파이어]](검증)의 쌍이며, [[버파이어]]가 루프 품질의 진짜 병목이다. 버파이어를 작성하는 것은 곧 '좋다'의 의미를 정의하는 보상 함수를 설계하는 것이다.

플랫폼 구현 사례로 Cloudflare Agents가 있다([[build-agents-on-cloudflare]] 참조). 각 에이전트 인스턴스는 **지속적 ID(durable identity)**·로컬 SQL 스토리지·예약 작업·복구 가능한 실행을 보장하며, Cloudflare 글로벌 네트워크에서 수천만 개의 인스턴스로 확장 가능하다. 별도 인프라 관리 없이 통신 채널·에이전트 하네스·SDK 런타임·툴의 4층 구조로 에이전틱 시스템을 구성한다.

**세션 간 연속성(Cross-Session Continuity)**: 에이전틱 AI가 수 시간~수 일에 걸친 장기 작업을 맡을 때, 새 세션은 이전 세션의 맥락 없이 시작되는 근본 문제가 있다. Anthropic 엔지니어링 팀은 이를 "교대 근무마다 전임자 기억 없이 출근하는 엔지니어"에 비유하며, 초기화 에이전트·코딩 에이전트 역할 분리 + git 히스토리·진행 파일·JSON 기능 목록 조합으로 해결하는 [[장기-에이전트-하네스]] 패턴을 제안한다. 이 패턴에서 한 세션에 기능 하나만 구현하는 원칙이 조기 완료 선언과 원샷 구현 실패를 구조적으로 방지한다.

**AI 개발 피드백 루프 병목**: 에이전틱 AI를 실제 제품 개발에 투입할 때 코드·화면 생성 자체는 병목이 아니다. 진짜 병목은 생성된 결과물이 제품 맥락에 맞는지 검증하고 다음 지시를 구성하는 피드백 루프다 — Agent는 입력된 문제를 빠르게 풀지만 제품 맥락을 지속적으로 붙잡고 자율적 피드백을 돌리지 못한다(버즈빌 버즈포스 FE 개발 사례). 多Agent 병렬 처리 시에는 각 작업 배경 재기억·판단·다음 지시 반복에 따른 **맥락 전환 비용(context switching overhead)**이 새로운 병목으로 나타난다. 실용적 완화 방법은 작업 지시 시 QA 가이드라인 문서 + 테스트 데이터 자동 생성을 함께 지시하는 것이며, 구조적 해결 방향은 [[루프-엔지니어링]]의 자율 검증([[버파이어]]) 구조다. 작업 지시 자체의 완성도를 높이기 위해 **STICC Framework**(Situation·Task·Intent·Concern·Calibration 5관점)를 활용하면 모호한 프롬프트 대비 작업물 완성도가 향상되고 엣지케이스 발견에도 유용하다. 개념 상세는 [[ai-개발-피드백루프]] 참조.

**프로덕션 에이전트의 실제 조건**: 에이전틱 AI가 미션 크리티컬 환경에 투입될 때 드러나는 복잡성은 데모 환경에서는 보이지 않는다. [[프로덕션-에이전트]] 개념이 이 간극을 정의한다. 팔란티어 DevCon 6은 이 문제를 온톨로지 기반 에이전트 스택(오케스트레이터·에이전트 엔진·AIP Evolve)으로 접근한 사례다. 핵심 통찰은 AI 최적화가 반드시 "더 나은 AI"를 의미하지 않는다는 점이다 — 탬파 병원(TGH) 사례에서 AIP Evolve는 GPT-4.1 호출의 84%를 결정론적 TypeScript 함수로 대체했고, Claude Sonnet을 GPT-5 Mini로 교체해 비용 68% 절감과 전문가 선호도 90%를 동시에 달성했다. "모델 교체·프롬프트 최적화·AI 자체 제거 모두 수단이며 목적은 실제 목표 가치 극대화"라는 팔란티어의 철학은, AI를 구현 세부사항으로 취급하는 실용주의적 에이전틱 AI 운영 관점을 대표한다.
