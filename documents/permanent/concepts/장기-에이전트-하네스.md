# 장기 에이전트 하네스 패턴

## 정의

여러 컨텍스트 윈도우에 걸쳐 일관된 진행을 유지하기 위해 에이전트 역할을 **초기화 에이전트**와 **코딩 에이전트**로 분리하고, git·진행 파일·JSON 기능 목록으로 세션 간 상태를 이어받는 에이전트 하네스 설계 패턴이다.

## 맥락

[[에이전틱-ai]] 시스템이 수 시간~수 일에 걸친 장기 작업을 맡을 때, 새 세션은 이전 세션의 맥락 없이 시작된다. 단순 compaction만으로는 에이전트가 현재 진행 상태를 정확히 파악하기 어렵고, 조기 완료 선언·원샷 구현 시도 같은 실패 패턴이 반복된다. 이 패턴은 두 역할로 책임을 분리해 문제를 해결한다.

**초기화 에이전트(Initializer Agent)** — 최초 세션에서만 실행:
- `init.sh`(개발 서버 실행 스크립트), `claude-progress.txt`(진행 로그), 초기 git 커밋, JSON 기능 목록 파일을 생성한다.

**코딩 에이전트(Coding Agent)** — 이후 모든 세션에서 실행:
- 세션 시작: git 로그 + 진행 파일 읽기 → 미완료 기능 하나 선택 → 서버 시작 → E2E 테스트 → 구현
- 세션 종료: 기능 목록 `passes` 필드 업데이트 → 커밋 → 진행 노트 갱신

기능 목록은 JSON 형식으로 관리한다 — Markdown 대비 에이전트가 임의로 수정·삭제할 가능성이 낮기 때문이다. 한 세션에 기능 하나만 구현하는 원칙이 조기 완료 선언과 원샷 구현 실패를 구조적으로 방지한다.

[[루프-엔지니어링]] 맥락에서 이 패턴은 '에이전트 친화적 코드베이스' 세 조건(가독성·실행 가능성·검증 가능성)을 하네스 설계에 내재화한 구현 사례다. 기능 완료 판정은 Puppeteer MCP 기반 브라우저 자동화([[버파이어]])가 담당해 에이전트의 자기 평가 편향을 방지한다.
