# AI와 함께 Salesforce 대체 CRM '버즈포스' 내재화 개발기

## 요약

버즈빌이 10년 이상 사용하던 Salesforce를 탈피해 자체 CRM '버즈포스(BuzzForce)'를 약 3개월 만에 개발·전환한 사례 기록이다. [[ax-ai-transformation|AX]] 실행 관점에서 AI Agent를 활용한 업무 규칙 역추적·FE 개발, 그리고 그 과정에서 드러난 [[ai-개발-피드백루프]] 병목이 핵심 논지다.

## 핵심 내용

### Salesforce 탈피 배경

버즈빌은 Salesforce를 **10년 이상** 사용했으나 전담팀 없이 외주사를 통해서만 커스터마이징이 가능했다. 자사 셀프서빙 플랫폼 광고센터 출시 후 Salesforce 연동 자동화가 추가되면서 세 시스템(광고센터·광고운영시스템·Salesforce) 간 정합성 유지 난이도가 올라갔고, **atomic 트랜잭션 구현이 사실상 불가**했다. AI Agent 도입 후에도 Salesforce 비효율이 병목이 되어 전체 업무 효율 개선이 미미했다. 7월 라이선스 연장을 4개월 앞두고 **연간 라이선스 비용 약 1억 원** 절약을 검토하면서 내재화 개발이 시작됐다.

### 불확실성을 낮추는 첫 걸음: ERD와 PoC

자체 CRM 개발은 되돌리기 어려운 결정이므로, 팀은 두 가지 산출물로 전환 가능성을 먼저 검증했다.

1. **인터랙티브 ERD**: AI Agent를 활용해 엔티티 클릭 시 관계를 하이라이트하고 업무 시나리오별 관여 엔티티를 보여주는 형태로 제작. 막연한 의심을 "세일즈포스의 이 데이터는 왜 버즈포스에서는 이렇게 저장하나요?"라는 구체적 질문으로 전환하는 역할을 했다.
2. **PoC**: 4월 말까지 실제 사용자에게 시연 후 완전 전환 여부 판단.

### Salesforce의 실제 역할 범위

버즈빌에서 Salesforce는 단순 영업 파이프라인 도구가 아니었다. MC팀의 Pardot 마케팅 운영, 세일즈팀의 **기회(Opportunity)** 관리(제안→체결→광고 운영→정산 전 사이클), 재무팀의 세금계산서 매칭·수취·지급 기록, 그리고 기회·클라이언트 중심 자유 리포트를 모두 포괄하는 **CRM + 마케팅 + 영업 리포트 + 정산 통합 시스템**이었다.

### AI 분석으로 업무 규칙 역추적 + 인터뷰로 검증

저자(Frank Koh)는 Salesforce를 직접 사용한 경험이 없었기 때문에 AI Agent로 Salesforce 데이터 패턴을 분석해 업무 규칙을 역추적하는 방법을 택했다.

- 기회가 체결 단계로 넘어가면 집행 시작일·종료일 채움률이 **20% → 89%**로 증가 → 버즈포스 Stage 전환 validation 규칙으로 직결
- 세팅요청 단계에서 Daily Cap·타겟팅 정보 채움률 **90% 이상** 확인
- 일부 대행사는 수십~수백 건 광고 집행 후 통합 세금계산서 1건 발행 → 기회별 수수료율 비중 기준 금액 분배 fallback rule 도출, `OppBilling` 별도 테이블 분리의 근거 확보

AI가 알 수 없는 것(Stage 역방향 이동 여부, 실제 입력 화면 순서, 재무팀 예외 케이스 처리 기준)은 세일즈팀·MC팀·재무팀 **사용자 인터뷰**로 검증했다.

> "AI는 분석의 속도를 높여주었지만, 그 분석이 실제 업무와 맞는지 확인하고 제품의 책임 범위로 바꾸는 과정에는 커뮤니케이션이 필요했습니다."

AI의 역할은 분석 속도 향상·가설 생성·인터뷰 질문 목록 작성이었으며, 가설 검증·제품 반영 규칙 결정·이해관계자 설득은 사람이 담당했다.

### 데이터 재설계: 명시성 중심

Salesforce에서 하나의 필드가 여러 의미를 갖는 구조가 데이터 유실 위험을 만들었다. 버즈포스는 **"명시성"**을 중심 원칙으로 관계 테이블을 적극 활용해 이를 해결했다.

1. **`OppAccount` 테이블**: 기존에는 광고주·대행사·미디어렙사를 기회 테이블 필드 레벨로 관리해 3개 이상 클라이언트 참여 시 데이터 유실이 발생했다. 별도 테이블로 분리해 각 연결에 `advertiser` / `agency` / `media_rep` **역할**과 수수료율을 저장한다.
2. **`AccountContact` 중간 테이블**: 한 담당자가 여러 광고주·대행사와 연결되는 N:M 관계를 `role`, `is_primary`, `is_active`로 명시화.
3. **`OppBilling` 테이블**: 대행사 통합 세금계산서 1건이 여러 기회에 분배될 수 있고, 장기 캠페인은 한 기회가 여러 번 정산될 수 있으므로 Opportunity ↔ Billing N:M 관계와 기회별 배정 금액을 함께 저장.
4. **예산 vs 소진액 명시적 분리**: 기존 Salesforce에서 미구분으로 인해 광고운영플랫폼 병행 조회가 필요했던 문제를 해소.

### AI 개발의 실제 병목: [[ai-개발-피드백루프]]

저자는 FE 개발 경험이 전무한 상태에서 AI Agent로 프론트엔드를 A-Z 개발했다. 기능 구현 자체는 병목이 아니었다. 진짜 병목은 **결과물 검증과 피드백 루프**였다. 상세 개념은 [[ai-개발-피드백루프]] 참조.

AI Agent는 입력된 문제를 빠르게 풀지만 제품 맥락을 스스로 붙잡지 못한다. FE 엔지니어라면 던질 질문("bulk 작업인데 단건 모달만 있으면 안 되지 않나?", "4XX 에러인데 에러 토스트가 불친절하지 않나?")을 Agent는 스스로 던지지 않는다.

이를 극복하기 위한 해결책:
- 작업 지시 시 **QA용 테스트 데이터 생성 + QA 가이드라인 HTML 문서**를 함께 작성하도록 지시. Agent가 staging DB에 접근해 데이터를 선택·생성하고, 가이드라인 문서는 비CLI 환경에서 지시 인터페이스로도 활용.
- 출시 후 여러 Agent를 동시에 돌려 이슈를 **병렬 처리**.

그러나 여러 Agent 동시 결과물 생성은 새로운 병목을 낳았다 — 각 작업 배경 재기억·판단·다음 지시 반복에 따른 **맥락 전환 비용(context switching)**:

> "마치 CPU가 실제 연산보다 context switching에 더 많은 시간을 쓰는 상태에 가까웠습니다."

향후 계획으로 Agent가 스스로 피드백 루프를 돌릴 수 있는 프레임워크 선행 구축을 제시한다 — 사용자 시나리오, acceptance criteria, 테스트 데이터를 Agent가 스스로 확인하는 방식으로. [[루프-엔지니어링]] 패러다임이 지향하는 자율 검증 루프와 같은 방향이다.

> "AI 개발의 다음 병목은 코드를 더 빨리 쓰는 것이 아니라 피드백 루프를 얼마나 자율적으로 돌릴 수 있느냐에 있다."

### STICC Framework 활용

**Situation·Task·Intent·Concern·Calibration** 5가지 관점으로 작업 지시를 구성하는 STICC Framework를 AI 개발에 적용했다. 모호한 프롬프트 대비 작업물 완성도가 향상되고 미처 생각하지 못한 엣지케이스 발견에도 유용했다고 밝혔다.

### 런칭 결과와 교훈

6월 15일 공식 런칭 후 세일즈팀·재무팀 모두 Salesforce 없이 기존 업무를 수행 중이다. 두 팀의 반응은 대조적이었다.

- **세일즈팀**: "기존 업무를 문제 없이 이어갈 수 있다" 수준. 버즈포스를 엔티티·데이터 중심으로 사용하며 Salesforce 밖에서 일어난 일을 기록·조회하는 시스템으로 인식.
- **재무팀**: "업무가 많이 효율화되었다" 명확한 반응. 매입 세금계산서-기회 매칭, 매출 세금계산서 업로드 등을 **행동·유스케이스 중심**으로 사용. 기존 Salesforce에서는 유스케이스가 기회 상세 페이지 곳곳에 흩어져 있었으나, 버즈포스는 유스케이스 단위로 페이지·URL을 분리하고 자동화 기능을 추가했다.

핵심 교훈:
- 업무 시스템 만족도는 **"사용자의 일을 어떤 단위로 제품화했는가"**에 크게 좌우된다.
- 비개발자도 자신의 업무를 유스케이스 단위로 정의할 수 있어야 Agent 기반 업무 효율화가 쉬워진다.
- **"모든 것을 의심하는 태도"**가 AX 시대의 중요한 덕목이며, 출발점은 "당연해서 의심하지 않았던 것이 뭘까?"다.
- AI와의 분업: AI = 빠른 구현·반복 작업·context switching 보조 / 사람 = 도메인 지식 전달·사용자 니즈 파악·설득·프로젝트 주도.
