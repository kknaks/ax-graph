# Controller·Service·Repository 계층 분리와 관심사 분리 원칙

## 요약

Spring 백엔드에서 Controller·Service·Repository를 왜 나눠야 하는지를 관심사 분리 원칙으로 설명하는 교육 영상이다. 계층 간 의존 방향·DTO 경계·흔한 안티패턴·실용적 분리 기준을 구체적인 코드 비교와 함께 다룬다. 개념 상세는 [[계층형-아키텍처]]로 위임한다.

## 핵심 내용

### 도입 — 식당 비유

주문받는 홀·요리하는 주방·재료 관리하는 창고가 나뉘듯, 소프트웨어도 코드가 각자 한 가지 일만 맡아야 한다는 직관적 비유로 [[계층형-아키텍처]]와 관심사 분리를 도입한다. 한 사람이 모든 일을 맡으면 주문이 밀리는 순간 주방도 창고도 멈추는 것처럼, 한 코드가 여러 관심사를 맡으면 한 곳의 변경이 전체를 흔든다.

### 관심사 분리(Separation of Concerns)

"한 코드 조각은 한 가지 관심사만 책임진다" — 그래야 바뀌는 이유도 하나가 된다. 대표 관심사 세 가지:

| 관심사 | 역할 |
|--------|------|
| 표현 | 요청을 받고 응답 형식을 만드는 일 |
| 비즈니스 | 핵심 규칙과 흐름을 처리하는 일 |
| 영속 | 데이터를 저장하고 조회하는 일 |

세 관심사가 뒤섞이면: (1) 할인 정책 하나를 바꾸려 해도 요청 처리·저장 코드까지 건드려야 하고, (2) 비즈니스 로직만 단독 테스트하고 싶어도 웹 요청·DB가 묶여 있어 테스트가 어려우며, (3) 같은 로직을 다른 곳에서 재사용할 수 없다.

### 계층형 아키텍처 기본 구조

클라이언트 요청 → **Controller** → **Service** → **Repository** → Database, 응답은 역순. [[계층형-아키텍처]]의 핵심 규칙은 "각 계층은 바로 아래 계층만 호출하고 계층을 건너뛰지 않는다"는 것이다. 컨트롤러가 서비스를 건너뛰고 리포지토리를 직접 호출하면 중간 계층을 독립적으로 교체할 수 없게 된다.

각 계층의 역할과 금지 사항:

| 계층 | 해야 할 일 | 하지 말아야 할 일 |
|------|-----------|------------------|
| Controller | 요청·응답 변환, 입력 검증 | 비즈니스 규칙 판단 |
| Service | 비즈니스 규칙, 트랜잭션 경계 | HTTP·SQL 직접 처리 |
| Repository | 데이터 저장·조회 | 비즈니스 판단 |

코드 배치 판단 기준: **"이 코드가 바뀌는 이유가 무엇인가?"** — 화면 형식 때문이면 Controller, 정책 때문이면 Service, 저장 방식 때문이면 Repository.

### 코드 예시 비교

**나쁜 예(뒤섞인 컨트롤러)**: 주문 생성 메서드 하나 안에 입력 검증·할인 비즈니스 규칙·DB INSERT가 혼재. 요청을 `Map`으로 받아 타입 안정성 없음. 할인 정책·저장 방식·요청 형식이 모두 바뀔 때 이 컨트롤러를 수정해야 한다.

**좋은 예(계층 분리)**:
- *Controller*: `@Valid`가 붙은 `OrderRequest`로 수신 → Command 객체 변환 → Service 위임 → 결과를 응답 객체로 변환. 컨트롤러는 **통역사** 역할만 하며, 할인 정책이 바뀌어도 수정 불필요.
- *Service*: `@Transactional`로 트랜잭션 경계 설정. `Order.create()` 호출로 핵심 규칙을 **도메인 객체** 안에 위치시키고 Service는 흐름 조율에만 집중 → 얇은 Service 유지.
- *Repository*: 데이터 저장소 접근을 추상화. Service는 `save()` 인터페이스만 알면 되므로 저장 기술 교체·테스트용 가짜 구현 주입·쿼리 변경이 비즈니스 로직에 영향을 주지 않는다. Spring Data JPA는 인터페이스 선언만으로 구현체를 자동 생성해 이 추상화를 거의 공짜로 제공한다.

### DTO 경계

데이터 흐름: 클라이언트 요청 → **요청 DTO** → 도메인 객체 → **응답 DTO** → 클라이언트. 엔티티를 외부에 직접 노출하면 내부 데이터 구조 노출과 비밀번호 같은 민감 필드 유출(보안 사고)이 발생한다. DTO는 계층 사이를 막아주는 **방어망**이다.

### 의존성 방향 규칙

세 원칙: (1) **단방향** — 위 계층이 아래 계층에만 의존, 역방향 금지; (2) **계층 건너뛰기 금지**; (3) **추상화 의존** — 구체 구현이 아닌 인터페이스에 의존. 가장 중요한 것은 단방향이다. Service가 Controller를 알면 비즈니스 로직이 특정 표현 방식(웹 화면)에 묶여 배치나 다른 진입점에서 재사용할 수 없게 된다.

### 계층화의 트레이드오프

| 장점 | 비용 |
|------|------|
| 변경이 한 계층에 격리됨 | 코드량 증가 |
| 테스트 용이 | 단순 기능도 여러 파일 필요 |
| 역할 명확 | 단순 CRUD에 기계적 적용 시 보일러플레이트만 증가 |

**실용적 기준**: 변경과 테스트가 필요할 만큼 복잡도가 생길 때 분리. 모든 코드를 똑같이 나눌 필요는 없다.

### 흔한 안티패턴

| 안티패턴 | 설명 | 해결책 |
|----------|------|--------|
| 뚱뚱한 컨트롤러 | 컨트롤러에 비즈니스 로직 | Service로 이동 |
| 빈약한 서비스 | Service가 단순 호출만 전달 | 도메인 객체에 규칙 부여 |
| 엔티티 노출 | 엔티티를 응답으로 직접 반환 | 응답 DTO로 변환 |
| 계층 침투 | Service 안에 HTTP 요청 객체 | 계층 전용 객체 사용 |
| 양방향 의존 | 아래 계층이 위 계층을 참조 | 단방향으로 정리 |

공통 원인: **한 계층이 자기 책임을 넘어서는 일을 함**.

### 실무 적용 팁

1. 코드를 짤 때 "이게 바뀌는 이유가 무엇인가"를 기준으로 계층 결정
2. 엔티티는 계층 밖으로 내보내지 말고 DTO로 변환해 반환
3. Service가 얇으면 도메인 객체에 규칙을 넣어 빈약한 서비스 방지
4. 단순 CRUD까지 기계적으로 나누지 말고 복잡도가 생길 때 분리
