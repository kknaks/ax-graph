# HITL 승인 위치 결정 패턴 — Mastra 에이전트·워크플로 가이드

## 요약

Mastra 프레임워크에서 Human-in-the-Loop 승인을 구현할 때 '언제'가 아닌 '어디에' 넣을지를 다룬 실전 가이드다. 에이전트가 진입점인 경우와 워크플로가 진입점인 경우 각각 두 가지 패턴을 제시하며, "위험이 어디에 있는가"와 "사람이 합리적으로 판단할 수 있는 시점이 언제인가"를 패턴 선택의 기준으로 제시한다.

## 핵심 내용

### 두 가지 승인 메커니즘

Mastra는 두 가지 승인 진입점을 지원한다.

- **에이전트**: 툴 호출 전 승인 요청 후 일시정지 → 사람 결정 대기 → 재개
- **워크플로**: 특정 스텝에서 `suspend()` 호출 후 resume 대기

이 글은 이전 포스트의 "언제(when)" 논의에 이어, 실제 구현 위치인 "어디서(where)"에 집중한다. 실행 흐름을 이해하는 핵심 프레임은 **Handoff Graph**다: 에이전트→툴→워크플로, 또는 워크플로→스텝→중첩 에이전트로 흐름이 이어지며, 사람이 개입 가능한 지점은 툴 호출 이전 또는 워크플로 스텝 내부다.

### 패턴 1 — 에이전트가 진입점일 때

에이전트가 먼저 실행되고 툴을 통해 워크플로를 시작하는 구조다. [[hitl-승인-패턴]]의 두 가지 변형이 있다.

**패턴 1a: 툴 레벨 승인**
- 에이전트가 툴을 호출하기 **전에** 승인 요청 → 승인 후 툴 실행 → 워크플로 시작
- 적합 상황: 툴 호출 자체가 위험한 행위일 때 (이메일 발송, 환불 처리, 결제 트리거)
- 트레이드오프: 워크플로는 승인이 발생했다는 사실을 알지 못하고, 워크플로 상태·실행 이력에 해당 일시정지가 기록되지 않는다

**패턴 1b: 워크플로 레벨 승인**
- 에이전트가 툴을 즉시 호출하고 워크플로 실행이 시작된다. 승인은 워크플로 내 특정 스텝에서 발생
- 적합 상황: 위험한 행위가 워크플로 스텝 내부에 있을 때. 컨텍스트를 충분히 수집한 뒤 승인 요청 가능
- 트레이드오프: 승인 요청 시점에 이미 툴은 실행된 상태이므로, 툴 호출 자체가 위험하다면 이 패턴으로는 막을 수 없다

### 패턴 2 — 워크플로가 진입점일 때

워크플로가 먼저 실행되고 내부에서 에이전트를 호출하는 구조다. [[agent-workflow-조합]] 관점에서 에이전트를 스텝으로 쓰는 방식이 여기에 해당한다.

**패턴 2a: 스텝 안에서 에이전트 호출**
- 워크플로 스텝이 일시정지 → 승인 후 재개 → `agent.generate()` 호출
- 적합 상황: 에이전트 호출을 세밀하게 제어해야 할 때 (동적 프롬프트 빌드, 메시지 이력 관리, 응답 변환)
- 트레이드오프: 에이전트 호출이 스텝 내부에 숨어 있어 로그·트레이스에서 워크플로 동작을 파악하기 어렵다

**패턴 2b: 에이전트를 스텝으로 구성 (`createStep`)**
- `createStep(agent)`으로 에이전트를 워크플로에 직접 스텝으로 삽입
- 승인 스텝이 `suspend()`로 일시정지 → 승인 후 `.map()`으로 출력을 `{ prompt }` 형태로 변환 → 에이전트 스텝 실행
- 적합 상황: 명확한 분리와 가시성이 필요할 때. 워크플로 정의에서 승인 스텝과 에이전트 스텝이 명시적으로 보임
- 트레이드오프: 스텝 수가 늘어나고 `.map()`으로 스텝 간 데이터 변환이 필요하다

### 패턴 선택 기준

저자가 제시하는 두 가지 판단 기준:

1. **위험이 어디에 있는가**: 툴 호출 자체가 위험 → 실행 전 승인(툴 레벨). 위험이 워크플로 스텝 내부 → 해당 스텝에서 승인
2. **사람이 합리적으로 판단할 수 있는 시점이 언제인가**: 충분한 컨텍스트가 수집된 시점에 승인을 요청해야 한다

추가 고려사항:
- **다층 승인(Layered approval)**: 넓은 결정과 특정 고위험 행위를 별도로 확인할 때 유용하지만, 복수의 승인 프롬프트는 사용자 피로를 유발할 수 있다
- **설계 재검토 신호**: 승인이 워크플로 상태에 의존하는데 아직 아무것도 실행되지 않았다면 설계를 재검토해야 한다는 신호다
