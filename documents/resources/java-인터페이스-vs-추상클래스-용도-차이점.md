# Java 인터페이스 vs 추상클래스 용도 차이점 완벽 이해

## 요약

Java의 인터페이스와 추상클래스는 기능 차이가 아닌 **사용 목적의 차이**로 구분된다. 추상클래스는 클래스 간 계층 관계 구축과 중복 멤버 통합에, 인터페이스는 상속 관계와 무관하게 자유롭게 타입을 묶고 동작을 보장하는 데 초점을 둔다.

## 핵심 내용

### 비교표

| 구분 | 추상 클래스 | 인터페이스 |
|------|------------|------------|
| 키워드 | `abstract` | `interface` |
| 사용 가능 변수 | 제한 없음 | `static final` (상수)만 |
| 접근 제어자 | 제한 없음 | `public`만 |
| 사용 가능 메소드 | 제한 없음 | abstract, default, static, private |
| 상속 키워드 | `extends` | `implements` |
| 다중 상속 | 불가능 | 가능 (다중 구현 + 인터페이스 간 다중 상속) |

공통점: 추상 메소드를 가져야 하며 인스턴스화 불가. 상속/구현한 클래스는 추상 메소드를 반드시 구현해야 한다.

### 인터페이스 — 자유로운 타입 묶음

[[java-인터페이스]] 참조. 핵심은 구현 객체들이 **같은 동작을 한다는 것을 보장**하기 위해 사용하며, 상속 계층과 무관하게 **자유롭게 타입을 묶는다**. `implements` 키워드처럼 각 클래스 목적에 맞게 기능을 구현하는 느낌이다.

인터페이스를 사용하는 경우:
- 서로 관련 없는 클래스들을 형제 타입으로 묶어야 할 때 (다중 구현)
- 특정 행동을 명시하되 구현 위치는 무관할 때
- 마커 인터페이스로 타입 체크만 필요할 때

Creature-Animal-Fish 계층에서 `swimming()`을 People과 Whale에만 추가할 때, 최상위에 추상 메소드를 넣으면 Tiger·Parrot에도 강제 구현이 필요해진다. `Swimmable`·`Flyable`·`Talkable` 인터페이스로 분리하면 각 클래스가 필요한 것만 구현한다. 인터페이스 타입을 중개 타입(예: `Storable target` 필드)으로 활용하면 전혀 다른 클래스들을 동일 타입으로 처리할 수 있다.

### 추상클래스 — 계층 구조와 중복 통합

[[java-추상클래스]] 참조. 하위 클래스들의 공통점을 모아 **클래스 간 연관 관계를 구축**하고 **중복 멤버를 통합·확장**하는 데 초점을 둔다. `extends` 키워드처럼 기능을 하위 클래스로 확장시키는 느낌이다.

추상클래스를 사용하는 경우:
- 공통 메소드·필드가 많아 중복 통합이 필요할 때
- `protected`/`private` 접근자나 non-static/non-final 필드가 필요할 때
- 삼각형·원→도형, 사자·고양이→동물처럼 명확한 계층 구조(is-a 관계)가 있을 때

SMS Sender 예시: 각 통신사는 커넥션 구현(`establishConnectionWithYourTower()`)이 다르지만, 공통 방해금지 모드 체크 로직은 추상클래스 `SMSSender`에서 공유한다. 클라이언트는 추상 클래스 타입으로 선언하고 업캐스팅으로 초기화해 다형성을 활용한다. 추상클래스는 클라이언트에서 자료형을 사용하기 전에 **미리 논리적인 클래스 상속 구조를 만들어 놓고 사용이 결정**되는 느낌이다.

### 마커 인터페이스

아무 메소드도 없는 빈 껍데기 인터페이스로, 객체의 **타입 정보만 제공**한다. `instanceof` 연산자로 타입을 구분하는 코드를 단순화하며, Java 표준 라이브러리의 `Serializable`·`Cloneable`이 대표 사례다.

### 인터페이스 + 추상클래스 조합 — Interface-Abstract-Concrete 패턴

인터페이스의 다중 구현 + 추상클래스의 중복 필드·메소드 통합을 동시에 활용하는 조합이다. 이 패턴 계열이 [[gof-디자인-패턴]]의 근간을 이룬다.

인터페이스만 쓰면 구현 클래스마다 공통 필드(`opacity`, `color`)가 중복된다. 중간에 추상클래스(`Shape`)를 두면 공통 부분을 통합하면서 `draw()`만 각 구체 클래스에 강제할 수 있다. 단점: 클래스 상속 기반이므로 이미 다른 클래스를 상속받아야 한다면 이 패턴을 활용할 수 없다 — 이 경우 [[구조-패턴]]의 Adapter 패턴을 대안으로 활용한다.
