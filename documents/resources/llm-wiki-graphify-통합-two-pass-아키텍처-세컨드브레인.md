# LLM 위키 × Graphify 통합과 Two-Pass 아키텍처로 만드는 세컨드 브레인

## 요약

[[llm-wiki]] 단독 운영이 문서 100개를 넘을 때 AI가 방향을 잃고 토큰 비용이 폭증하는 문제를, LLM 위키와 Graphify를 결합하고 [[two-pass-아키텍처]]로 AI 환각을 원천 차단해 해결하는 전략을 제시한 영상이다. 실제 프로덕션 사례에서 세션당 토큰 71.5배 절감이 보고된다.

## 핵심 내용

### AI 지식 위키의 실제 문제

공들여 만든 [[llm-wiki]]는 문서가 100개를 넘는 순간 AI가 길을 잃고 **토큰 비용만 낭비**되는 빈 껍데기로 전락한다. 문서 하나당 위키 문서 하나를 기계적으로 1:1 복사하는 실수가 빈번한데, 이 경우 지식이 유기적으로 연결되지 않고 방대한 텍스트 파일의 무덤이 된다. AI의 미세한 오류·환각이 위키 네트워크 전체에 기정 사실처럼 전파되는 것도 치명적 약점이다.

### Graphify 단독 사용의 한계

Graphify는 노드와 구조를 연결하는 데 강점을 가지지만, 노드 안의 내용이 부실하면 뼈대만 앙상한 빈약한 지식망이 된다. 더 치명적인 약점은 **비슷한 단어·동의어를 제대로 필터링하지 못해 노드들이 연결되지 못하는 고립 노드 현상**이다.

### LLM 위키 × Graphify 통합 전략

두 도구의 역할을 명확히 분리한 뒤 상호 보완한다.

- **LLM 위키**: 풍부한 '살코기' — 상세한 맥락과 텍스트를 꽉 채움
- **Graphify**: 단단한 '뼈대' — 구조적 그래프로 관계를 정의

질문 시 워크플로는 Graphify의 그래프 기반으로 빠르고 정확하게 위치를 탐색한 뒤, 세부 내용은 [[llm-wiki]]에서 깊이 있게 끌어오는 구조다.

**에이전트 AI의 고립 노드 해결**: Graphify 고립 노드 문제는 에이전트 AI가 해결한다. 로컬 환경에서 '영상'과 '수면'처럼 물리적 링크가 끊어진 키워드들의 숨겨진 의미론적 연관성을 자동 추론해 완전한 시맨틱 연결망을 구축한다.

### 토큰 비용 절감 — 실제 사례

위키 문서가 방대해지면 질문 하나당 수만 토큰의 텍스트를 통째로 읽어와야 한다. 실제 프로덕션 SaaS **북제로(bookzero).ai**의 시니어 엔지니어 에릭 사례:

| 방식 | 세션당 토큰 |
|---|---|
| 기존 RAG/위키 방식 | 20,000 토큰 |
| Graphify 지식 그래프 사전 컴파일 적용 | 280 토큰 |

절감 효과: **71.5배**.

### [[two-pass-아키텍처]] — AI 환각 차단

텍스트를 통째로 던져 위키·그래프 생성을 맡기면 100% 확률로 AI 환각이 발생한다. [[two-pass-아키텍처]]로 이를 원천 차단한다.

1. **1단계 (결정론적 뼈대 추출)**: 로컬에서 **Tree-sitter** 구문 분석기만으로 기계적·결정론적 뼈대를 추출 — AI 개입 없음. "초등학생이 설명서대로 레고 블록을 실수 없이 조립하는 것"에 비유된다.
2. **2단계 (AI 문맥 연결)**: 팩트 기반 뼈대 위에서 AI가 문맥을 부드럽고 촘촘하게 연결.

100% 팩트 기반 뼈대 위에 지식을 쌓으므로 환각이 끼어들 틈이 차단된다.

### 3단계 자동화 파이프라인

1. **raw 폴더**에 새로운 메모·스크랩을 자유롭게 투입
2. **ingest 스크립트 실행** → wiki 폴더에 주제별로 자동 누적·병합
3. **sync 명령어 실행** → 로컬 파일을 스캔해 정교한 그래프 파일로 빌드

사용자가 할 일은 자료를 던져주고 명령어 몇 번 입력하는 것뿐이다. 나머지는 지식의 복리로 자동으로 굴러가는 파이프라인이다.
