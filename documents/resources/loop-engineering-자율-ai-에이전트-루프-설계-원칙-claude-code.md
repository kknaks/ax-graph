# Loop Engineering: 자율 AI 에이전트 루프 설계 원칙 (Claude Code)

## 요약

Anthropic Claude Code 책임자가 "나는 더 이상 Claude에 직접 프롬프트하지 않는다. 루프가 대신 한다"고 공언하며 2026년 6월 공식화된 [[루프-엔지니어링]] 패러다임을 해설한 영상이다. 개발자가 에이전트를 직접 프롬프트하는 대신 에이전트를 구동하는 루프 시스템을 설계하며, 제너레이터와 [[버파이어]]의 쌍이 루프의 핵심 구조다.

## 핵심 내용

### 프롬프트 시대의 종말과 루프 엔지니어링의 등장

에이전트 40개가 풀 리퀘스트 3개를 남겨놓는 장면의 출발점은 프롬프트 40개가 아니라 **루프 하나**였다. Claude Code 책임자(Anthropic)의 발언이 이 패러다임 전환을 압축한다:

> "I do not prompt Claude anymore. I have loops running that prompt Claude and figure out what to do. My job is to write loops."

2026년 6월 한 주 만에 '루프 엔지니어링(Loop Engineering)'이라는 이름과 12페이지 분량의 현장 가이드가 등장해 확산됐다.

[[루프-엔지니어링]]은 개발자가 에이전트를 직접 프롬프트하는 내부 사이클에서 한 단계 올라가 에이전트가 달리는 트랙(시스템)을 설계하는 것이다. AI 작업 패러다임의 연대기: 2024년 운영자(프롬프트 작성) → 2025년 관리자(여러 에이전트 동시 운영) → 2026년 시스템 설계자(에이전트를 구동하는 시스템 설계).

스택 구조상 루프 엔지니어링은 단일 에이전트 실행에 필요한 모든 것(프롬프트·컨텍스트·하네스)을 타이머 위에서 구동하는 외부 시스템으로, 하네스보다 한 층 위에 위치한다.

### 루프의 5가지 핵심 요소 + 메모리

루프는 아래 5가지 요소와 메모리로 구성된다. [[에이전틱-ai]] 패러다임의 실전 구현체이며, 루프의 4단계 반복 구조는 Discover → Plan → Execute → Verify → 조건 충족 시 종료, 아니면 반복이다.

**1. 자동화(Automations) — 심장박동**: 스케줄에 따라 작업을 자동 발견·분류(triage)하고 인박스에 적재한다. 이것이 루프를 '한 번의 실행'이 아닌 진짜 루프로 만드는 요소다.

**2. 워크트리(Worktrees)**: 에이전트 2개 이상 동시 실행 시 파일 충돌을 막기 위해 Git 워크트리가 각 에이전트에 독립 체크아웃·독립 브랜치를 제공한다.

**3. 스킬(Skills)**: 프로젝트 지식(컨벤션, 빌드 절차, 특수 규칙 등)을 한 번 문서화한 것. 없으면 루프가 매 사이클마다 프로젝트 전체를 처음부터 재파생한다.

**4. 커넥터(Connectors) — MCP 기반**: MCP(Model Context Protocol)로 실제 도구(이슈 트래커, DB, Slack, Stripe 등)에 연결한다. "수정 방법을 말하는 에이전트" vs. "풀 리퀘스트를 직접 여는 루프"의 차이가 여기서 나온다.

**5. 서브에이전트(Sub-agents)**: 작성자와 검토자를 분리한다. 코드를 쓴 모델은 자신의 코드를 채점할 때 너무 관대하므로, 다른 지시를 받은 별도 에이전트가 첫 번째가 납득시킨 오류를 포착한다.

**메모리(Memory) — 척추**: 마크다운 파일, Linear 보드 등 대화 외부에 존재하는 것. 에이전트는 실행 간 모든 것을 잊지만 레포는 잊지 않는다. 완료된 것·다음 할 것을 기록하는 메모리가 루프 전체의 연속성을 유지한다.

### 제너레이터 vs. [[버파이어]]: 진짜 병목

루프 = 제너레이터(Generator) + [[버파이어]](Verifier)다.

- **제너레이터**: 작업을 생산하는 모델 — 현재 모델들은 이미 매우 뛰어남
- **[[버파이어]]**: 작업이 충분히 좋은지 판단하는 것 — **진짜 병목**

약한 [[버파이어]]의 결과가 **슬롭 머신(slop machine)**이다. "랜딩 페이지를 더 좋게 만들어. 계속해."라는 지시에 '더 좋다'의 정의가 없으면 히어로 섹션을 8번 다르게 재작성하고도 성공을 보고한다. 실제 돈을 지불하고 진전 없는 움직임만 얻는다.

[[버파이어]]를 작성하는 것은 보상 함수(reward function)를 정의하는 것이다. 모델을 훈련하는 게 아니라 '좋다'의 의미를 정의하는 것이며, 개발자의 취향·판단·올바름에 대한 감각이 해자(moat)가 된다. 모델은 범용 상품(commodity)이므로 에이전트가 자신의 작업을 스스로 채점하게 두면 안 된다. Claude Code의 `goal` 명령은 종료 결정을 별도의 빠른 모델에 위임해 코드를 쓴 모델이 완료를 선언하지 않도록 한다.

### 닫힌 루프 vs. 열린 루프

| 구분 | 닫힌 루프(Closed-Loop) | 열린 루프(Open-Loop) |
|------|------------------------|----------------------|
| 특징 | 성공 조건을 사전에 고정, 매 단계 검사 | 목표와 탐색 여지만 부여 |
| 장점 | 예측 가능, 저비용, 방치 안전 | 새로운 해결책 발견 가능 |
| 단점 | 창의적 탐색 제한 | 토큰 소모 많음, 슬롭 위험 |

열린 루프 구제 트릭: 하드 체크를 바닥으로 유지하면서 열린 지시 하나 추가("헤드라인은 놀라게 해줘"). 탐색은 허용하되 기준 이하로 내려갈 수 없다. [[버파이어]]가 두 종류 모두를 실제로 배포 가능하게 만드는 핵심이다.

### 자율성의 사다리

인프라를 처음부터 만들 필요 없는 명확한 단계:
1. 베어 셸 루프 — 개발자가 정지 버튼
2. `goal` 명령 — 소형 모델이 '완료' 판단
3. 목표 추적 설정
4. 항상 켜진(never-sleep) 에이전트

**원칙: 작업을 수행하는 가장 덜 자율적인 도구를 선택하라.**

### 에이전트 친화적 코드베이스 조건

루프가 작동하려면 코드베이스가 준비되어 있어야 한다:
- **가독성(Legible)**: 에이전트가 무엇을 변경할지 찾을 수 있도록
- **실행 가능성(Executable)**: 개발 서버가 이미 켜져 있어야
- **검증 가능성(Verifiable)**: 작업의 증거를 제시할 수 있어야(예: 브라우저 테스트가 클립을 녹화)

### 루프의 복리 효과와 한계

루프 A(지원 티켓 처리 + 학습 기록), 루프 B(콘텐츠 작성), 루프 C(로드맵 우선순위)가 같은 파일을 읽고 쓰면 한 루프가 배운 것을 다른 루프가 흡수한다. 한 개발자가 이 시스템으로 하루 20~40페이지를 직접 감시 없이 배포했다.

그러나 루프가 해결하지 못하는 세 가지 문제:
1. **검증은 여전히 개발자의 몫** — '완료'는 주장일 뿐, 증명이 아님
2. **이해력 부식** — 루프가 만든 것을 읽기를 멈추면 이해력이 썩음
3. **편안한 함정** — 루프가 주는 것을 그냥 받아들이는 것이 위험한 선택

### 최종 판정: 유행어인가, 실질적 전환인가?

용어 자체는 주목을 끌기 위해 만들어졌고 에이전트 빌더들이 이미 알던 내용을 일부 재포장한 것이지만, 전환 자체는 실재한다. 결정적 차이는 루프를 자신이 깊이 이해하는 작업에 더 빠르게 활용하는 사람과 루프를 이용해 작업 이해 자체를 회피하는 사람 사이에 있다. 루프는 그 차이를 모르지만 개발자는 안다.
