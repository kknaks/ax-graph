# MVCC(다중 버전 동시성 제어) 개념과 동작 원리

## 요약

MVCC의 탄생 배경(락 방식의 한계)부터 타임스탬프 기반 버전 관리 원리, SCN·언두 영역·CR카피 구성 요소, 그리고 자원 오버헤드 트레이드오프까지를 한 강의에서 다루는 입문 영상이다. 실무에서 쿼리 성능 저하를 진단할 때 "언두 블록 I/O 과부하"로 원인을 구체화하는 데 필요한 개념 지형을 제공한다.

## 핵심 내용

### 동시성 문제와 전통적 해결책 — 락(Lock)

수십~수천 명이 동시에 데이터베이스에 접속하면 동일 데이터를 여러 사용자가 동시에 수정해 **데이터 부정합**이 발생한다. 기존 해결책인 **락(Lock)**은 한 사용자가 수정하는 동안 다른 사용자의 접근을 차단함으로써 정합성을 보장하지만, 대기 줄이 생겨 전체 시스템 처리 속도가 낮아지는 단점이 있다.

### MVCC 개요 — 읽기와 쓰기의 분리

[[mvcc]]는 락 대신 데이터의 여러 버전을 관리하여 읽는 사람과 쓰는 사람이 서로 방해하지 않고 각자 작업할 수 있게 하는 동시성 제어 기법이다. 개념 상세는 [[mvcc]] 참조.

### 동작 원리 — 타임스탬프 기반 버전 관리

모든 트랜잭션은 시작 시 고유한 **타임스탬프**를 부여받고, 데이터베이스 내 각 데이터도 버전별로 타임스탬프를 가진다. 트랜잭션이 데이터를 읽을 때 시스템은 두 타임스탬프를 비교해 해당 시점에 맞는 버전을 제공한다. 결과적으로 각 트랜잭션은 자신이 시작된 순간의 **스냅샷**을 독립적으로 조회해 다른 사용자의 변경에 영향받지 않는다. 이를 통해 모든 작업이 순서대로 직렬 실행되는 것처럼 보이는 **직렬 가능성(Serializability)**이 보장된다.

> 강의 비유: "MVCC는 똑똑한 도서관 사서 같은 것. 작가가 원본을 수정하는 동안, 독자들에게는 수정 전 복사본을 나눠줘 서로 방해받지 않게 한다."

### 내부 동작 단계 — 업데이트 시 흐름

1. 업데이트 요청 수신
2. 기존 데이터를 **언두(Undo) 영역**(임시 대피소)에 백업
3. 원래 공간을 새로운 데이터로 업데이트
4. 사용자가 **커밋(Commit)** 하면 **SCN(System Change Number)**이 1 증가

이 과정 동안 다른 사용자들은 언두 영역의 이전 버전 데이터를 간섭 없이 계속 읽을 수 있다.

### 핵심 구성 요소

- **SCN(System Change Number)**: 시스템 전체의 시계 역할. 트랜잭션이 커밋될 때마다 증가.
- **언두(Undo) 영역**: 데이터가 변경될 때마다 이전 버전을 보관하는 저장소(도서관 비유의 '서고').
- **CR카피(Consistent Read Copy)**: 일관된 읽기를 위한 복사본. SCN을 기준으로 쿼리가 요청한 시점의 데이터 버전을 언두 영역에서 찾아 제공.

### 트레이드오프 — MVCC의 비용

높은 동시성을 얻는 대신 시스템 자원을 추가로 소비한다.

| 비용 유형 | 설명 |
|---|---|
| **I/O 오버헤드** | 오래된 데이터를 읽기 위해 언두 영역까지 디스크 접근 증가 |
| **스토리지 오버헤드** | 데이터의 이전 버전들을 계속 보관하므로 저장 공간 추가 소요 |
| **CPU 오버헤드** | 버전 관리 및 버전 선택 계산으로 CPU 부하 증가 |

### 실무적 시사점

읽기 동시성과 쓰기 처리량은 매우 높고, 자원 비용은 중간~높음 수준이다. 서비스 성능 저하 시 단순히 "느리다"에서 나아가 "이 쿼리가 오래된 데이터를 찾느라 언두 블록을 계속 뒤지며 I/O가 과부하되는 것은 아닌가"라는 구체적 진단이 가능해진다. Oracle, PostgreSQL 등 현대 데이터베이스 대부분이 MVCC를 채택하고 있으며, 내부 버전 관리가 시스템 성능에 미치는 영향을 이해하는 것이 중요하다.
