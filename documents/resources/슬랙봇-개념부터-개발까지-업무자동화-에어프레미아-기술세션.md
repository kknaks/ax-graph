# 슬랙봇 개념부터 개발까지 — 업무 자동화 슬랙봇 제작 가이드 (에어프레미아 기술 세션)

## 요약

에어프레미아 이승민(maryang)이 2024년 5월 사내 기술 세션에서 발표한 슬랙봇 입문·실전 가이드다. 봇 개념과 슬랙봇 정의부터 슬랙 앱 생성·토큰 발급·서버 개발 방식 선택·설정 상세까지 전 과정을 다루며, 사내 실용 예시 세 가지를 통해 전사 생산성 향상 효과를 보여 준다.

## 핵심 내용

### 봇과 슬랙봇 정의

봇은 "반복적인 작업을 사람 대신에 수행하는 소프트웨어 프로그램"이다. GitHub Actions(배포 자동화), Jira Automation(PR 연결 시 티켓 업데이트), CS 챗봇이 대표 사례다.

슬랙봇은 발표자 정의로 **슬랙과 인터랙션하는 봇**이다. 봇 자체와 슬랙은 별개이며, 슬랙은 봇의 입출력 인터페이스 역할을 한다. 방향은 두 가지다: ① 외부 이벤트 → 슬랙 알림 수신, ② 슬랙 명령 → 외부 도구 명령 수행. 실제 사례로 Polly(설문), Donut(랜덤 미팅), Jira 슬랙봇(알림+이슈 생성), 뱅크샐러드(슬랙으로 배포 명령·배포 상태 조회)가 언급된다.

### 왜 슬랙봇인가

전 사원이 압도적으로 가장 오래 쓰는 툴이 메신저(슬랙)이므로, 항상 켜진 이 허브에서 알림 수신과 외부 도구 조작을 통합하면 개발·비개발 공통으로 **전사 생산성 향상**이 가능하다.

### 슬랙 앱 생성 절차

1. [api.slack.com](https://api.slack.com)에서 `Create New App` → **From scratch** 선택 (App Manifest는 JSON 직접 작성 필요로 어렵다)
2. 앱 이름·워크스페이스 설정 → 슬랙 클라우드에 업로드(아직 설치 아님)
3. **Features & Functionality** → **Bots** → App Display Name 설정 → 봇 유저 생성
4. **Install your app** → 워크스페이스 설치 → **Bot User OAuth Token** 발급 완료

Bot Token 발급 = 슬랙 이벤트 수신 준비 완료.

### 봇 서버 개발 방식 세 가지

세 가지 방식의 선택 기준과 상세는 [[슬랙봇-개발-방식]] 참조.

간략 요지:
- **Incoming Webhook**: 코드 거의 없이 알림 전용. Webhook URL 호출 → 슬랙 메시지 전송. 양방향 불가.
- **Bolt 프레임워크**: 슬랙 소통에 최적화된 서버 프레임워크(JS/Python/Java). 슬랙 내 다양한 사용자 동작 리스닝 가능. 구현 편의성 높으나 서버 프레임워크 자유도 낮음.
- **Slack SDK**: 슬랙 소통 기능 패키지. 서버 프레임워크는 별도 선택(NestJS, Django 등). 핵심 패키지 Web API가 130개 이상 메소드 제공. 발표자는 **서버 커스터마이징 자유도**를 이유로 SDK + NestJS 선택.

### 설정 상세

**토큰 권한(Scopes)**: 메소드 사용 전 공식 문서에서 Required scopes 확인 필수. 예: `chat.postMessage` 사용 시 `chat:write` 스코프가 있어야 하며, 콘솔 → OAuth & Permissions → Scopes에서 추가 후 재설치한다.

**이벤트 구독(Event Subscriptions)**: 콘솔 → Event Subscriptions → ON → 이벤트 추가(`app_mention`, `message.channels` 등) → 배포된 서버 도메인을 Request URL로 입력 → 이벤트 발생 시 해당 URL 호출.

**소켓 모드(Socket Mode)**: 기업 내부망처럼 슬랙이 서버에 직접 호출할 수 없는 환경에서 사용. 콘솔 → Socket Mode → ON → App-Level Token 별도 생성 → `SocketModeClient` 사용(기존 `WebClient`와 다름). URL 노출 없이 슬랙 앱과 서버를 직접 연결한다.

> 레퍼런스를 자주 봐야 하는 세 가지: 받고 싶은 이벤트 이름, 쓰고 싶은 메소드 이름, 필요한 권한(Scope).

### 사내 활용 예시

**예시 1 — 휴가 인원 자동 알림**: 매일 오전 9시 스케줄러 → 인사 DB 조회 → 슬랙봇 서버 → 특정 채널 알림. 별도 문의 없이 자동 수신.

**예시 2 — 항공 예약률 조회**: 직원이 슬랙 채팅으로 "5월 7일 NRT 예약률 알려줘" 입력 → 봇이 예약 DB/Tableau 조회 → 결과 회신. DB·Tableau 접근 권한 없는 비개발 직원(승무원, 정비사)도 사용 가능 → **접근성 민주화** 효과.

**예시 3 — 비행 채널 자동 생성·아카이브**: 스케줄러가 내일 비행 스케줄 조회 → 채널 자동 생성 + 조종사·승무원 자동 초대 → 비행 중 소통 → 비행 1~2일 후 채널 아카이브 + 평가 링크 발송. 채널 생성·초대·아카이브·히스토리 전 과정 자동화.

### 마무리 — 슬랙봇 기획 관점

- 슬랙봇 영향력은 개발팀뿐 아니라 전 사원으로 확산 → 전사 생산성 향상 레버
- 무엇을 만들지 막막할 때 출발점: "우리 업무 중 자동화할 수 있는 일이 무엇인가?"
- 기획 단계부터 다양한 이해관계자 의견 수렴 필요
- 스타트업 초창기가 아니면 얻기 어려운 경험(전사 생산성 기여, DB 접근 권한, 커리어 성과)임을 발표자가 강조
