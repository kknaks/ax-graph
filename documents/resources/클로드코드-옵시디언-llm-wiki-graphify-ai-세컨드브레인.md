# 클로드 코드×옵시디언×LLM Wiki×Graphify로 AI 세컨드 브레인 구축하기

## 요약

안드레 카파시가 제안한 [[llm-wiki]] 개념을 기반으로, 클로드 코드·옵시디언·Graphify를 조합해 AI 세컨드 브레인을 처음부터 세팅하는 전 과정을 실연한 영상이다. CLAUDE.md만으로는 대화 종료 후 지식이 사라지는 한계를 극복하기 위해 목적 있는 수집→AI 인제스트→위키 축적→그래프 연결의 복리형 지식 루프를 구성한다.

## 핵심 내용

### 문제 — CLAUDE.md만으로는 부족한 이유

클로드 코드 등 AI 에이전트 도구를 쓸 때 보통 `CLAUDE.md`(또는 `agent.md`, `gemini.md`)로 세팅하지만, 대화가 끝나면 지식이 사라진다. 재활용 가능한 나의 정보·지식을 축적하기 어렵고, AI가 나의 관점과 가치관을 지속적으로 파악하기 힘들다. 세컨드 브레인이 필요하지만 체계를 만들다 포기하는 경우가 많다.

### 해결 조합 — 왜 옵시디언인가

**옵시디언**은 순정 마크다운 기반 지식 관리 앱으로 AI 도구가 가장 잘 소화하는 포맷이다. 안드레 카파시(OpenAI 공동 창립자)가 2026-04-02경 옵시디언을 "AI를 위한 LLM 프런트엔드"라고 언급하며 폭발적 관심이 쏠렸다. 흐름은 세계 관찰 → 옵시디언 마크다운 입력 → 클로드 코드 프로세싱 → 구조화 문서(MD) → DOCX·PPT·PDF·MP4 등 다양한 산출물이다.

### Gold In, Gold Out — 목적 있는 수집

목적 있는 수집이 전제다. 내가 왜 수집했는지 설명할 수 없으면 진짜 의도적 수집이 아니다. 목적 없이 수집하면 쓰레기 데이터 → 산출물 품질 저하(Garbage In, Garbage Out). 저자는 이를 **"Gold In, Gold Out"**으로 3년 전부터 명명한다. 목적 있는 수집이 쌓여야 AI가 활용할 수 있는 골드 데이터가 늘어난다.

### [[llm-wiki]] — 개념과 RAG 비교

LLM Wiki는 안드레 카파시가 2026-04-03 트윗으로 제안한 지식 관리 프레임워크로, 바이브 코딩(2025-02), 컨텍스트 엔지니어링(2025-06)에 이어 세 번째 메타 트렌드로 부상했다. 개념 정의·핵심 흐름·스키마 상세는 [[llm-wiki]]에 위임한다.

RAG 대비 핵심 차이는 아래와 같다:

| 항목 | RAG | LLM Wiki |
|------|-----|----------|
| 셋업 복잡도 | 높음 (임베딩 모델, 벡터 DB 필요) | 낮음 (마크다운 파일만) |
| 인프라 | 벡터 DB 필수 | 로컬 폴더면 충분 |
| 검색 신뢰도 | 문서 품질·임베딩에 의존, 누락 발생 가능 | 목차·스키마 기반으로 상대적으로 높음 |
| 지식 축적 방식 | 문서 추가 시 재임베딩 필요 | 인제스트 시 위키에 연결·누적 |
| 복리 효과 | 상대적으로 약함 | **복리형** — 연결이 쌓일수록 가치 증가 |

### 세팅 실연 — 단계별 과정

#### 1단계: 볼트 생성 & 핵심 맥락 작성
새 볼트를 만든 뒤, STT로 **나는 누구인가 / 왜 기록하고 싶은가 / 어떤 아웃풋을 만들고 싶은가** 세 가지에 답해 핵심 맥락 노트를 작성한다. 이후 `"핵심 맥락 파일을 바탕으로 CLAUDE.md를 만들어줘(작업 규칙, 나는 누구인가, 기대 사항 등 포함)"` 프롬프트로 AI가 나의 목적·역할·작업 규칙을 포함한 CLAUDE.md를 자동 생성한다.

#### 2단계: 폴더 구조 & 위키 스키마
카파시 트윗 전문을 프롬프트에 붙여 나의 목적에 맞는 폴더 구조를 생성 요청한다. 결과물: `raw`(수집 소스), `wiki`(정리된 지식), `output`(산출물) 세 폴더와 각 폴더별 `CLAUDE.md`. 폴더별 `CLAUDE.md`로 AI에게 해당 영역의 맥락을 추가 제공한다.

#### 3단계: 웹 클리퍼 템플릿 커스텀
옵시디언 웹 클리퍼(크롬 익스텐션)의 기본 템플릿을 JSON으로 내보내 클로드 코드에 제공한다. LLM Wiki 스키마에 맞는 **아티클·유튜브·팟캐스트·책·연구자료** 5종 커스텀 템플릿을 생성해 웹 클리퍼에 임포트하면 사이트별 자동 트리거가 설정된다. 유튜브 영상 클리핑 시 트랜스크립트까지 자동 수집된다.

#### 4단계: 인제스트 — 목적성 있는 지식 소화
인제스트 전 내가 왜 이 자료를 수집했는지 코멘트 입력이 필수다(예: "LLM Wiki를 공부하고 브레인 트리니티 프로그램과 연결하고 싶어서 수집"). AI가 소스를 읽고 엔티티(LLM, RAG, Codex 등)를 분리해 기존 위키와 연결하고, `wiki` 폴더에 개념 문서를 생성한다. 대량 수집 시 클로드 프로 플랜 토큰 소모에 주의한다.

#### 5단계: 스킬 생성 & 워크플로우 자동화
인제스트 2회 이상 수행 후 AI가 패턴을 학습하면 `/ingest` 스킬을 생성 요청한다. 추가 스킬: `/query`(위키 내 마크다운 기반 검색 답변, 벡터 DB 없이), `/lint`(위키 전체 점검·최신화·깨진 링크 수정). 슬래시 커맨드 하나로 대화 없이 자동 실행된다.

#### 6단계: 위키 쿼리 & 신테시스
`/query "브레인 트리니티와 LLM 위키의 상관성 알려줘"` → 위키 내 관련 문서 검색 후 답변. 쿼리 결과를 `synthesis` 폴더에 저장하면 여러 소스를 통합한 보고서가 된다. 신테시스는 직접 활용이 아닌 새로운 지식 창출의 재료다.

### Graphify — 마크다운을 그래프 DB로

LLM Wiki는 문서가 많아질수록 키워드 기반 텍스트 검색 효율이 저하된다. 옵시디언의 그래프 뷰는 시각화 용도이며 AI가 그래프 정보를 직접 활용하지 않는다.

**Graphify**는 카파시 LLM Wiki 트윗 후 48시간 내에 개발된 오픈소스 도구로, **마크다운 위키를 그래프 DB로 변환**해 AI 코딩 어시스턴트가 노드·엣지 기반으로 더 정교한 쿼리를 수행할 수 있게 한다("Knowledge graphs for AI coding assistants").

```bash
pip install graphify
graphify wiki                              # 그래프 생성
graphify wiki --update                     # 변경된 부분만 재추출
graphify query "브레인 트리니티와 LLM의 근본적 차이"
```

출력물: `graph.json`(AI가 활용하는 그래프 데이터), `graph.html`(노드·엣지·커뮤니티 클러스터 시각화), 마크다운 리포트(옵시디언 내 확인 가능). 텍스트 기반 검색 대비 노드·엣지 관계를 활용한 더 정교한 연결 파악이 가능하다. 예시 쿼리 답변: "같은 문제를 다른 레이어에서 푸는 관계, 추상화 수준과 관심사 방향이 다름".

### 핵심 원칙

> "LLM이 알아서 해 주는 게 아닙니다. 제일 중요한 건 나의 목적입니다. 목적 없이 왜 수집했는지 답할 수 없으면 LLM Wiki를 제대로 활용하는 게 아닙니다."

이 조합(옵시디언 + 클로드 코드 + LLM Wiki + Graphify)이 현재 메타로 부상 중이며, 수집·연결이 쌓일수록 AI가 더 정밀하게 활용할 수 있는 **복리 효과**가 이 시스템의 핵심 가치다.
