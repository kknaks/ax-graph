# 파티셔닝 vs 샤딩: 대용량 데이터 분리 전략 비교

## 요약

데이터 폭증으로 단일 테이블·단일 DB가 한계에 직면할 때 적용하는 두 가지 분리 전략인 [[파티셔닝]]과 [[샤딩]]을 구체적인 비교 기준과 실무 사례로 해설한 영상이다. 둘의 핵심 차이는 물리적 위치(단일 vs 다중 인스턴스), 확장 방향(스케일업 vs 스케일아웃), 관리 주체(DB 엔진 vs 애플리케이션)로 요약된다.

## 핵심 내용

### 배경: 데이터 폭증이 만드는 네 가지 위기

- **IO 병목 심화**: 데이터 증가 → 인덱스 깊이 증가 → 디스크 접근 비용 상승
- **동시성 충돌 증가**: 대용량 트랜잭션 경합·락 대기 시간 증가
- **운영 비용 폭증**: 백업·복구·배포 시간 급증
- **물리적 서버 한계 도달**: CPU·메모리·디스크 단순 확장만으로는 불충분

이 네 위기의 해결책으로 데이터를 논리적(파티셔닝) 또는 물리적(샤딩)으로 분리하는 두 전략이 등장한다.

### 파티셔닝 — 단일 인스턴스 내부의 논리 분리

"한 지붕 아래 여러 개의 방을 만드는 개념"으로, 단일 DB 서버 내부에서 하나의 거대한 테이블을 여러 파티션(저장 단위)으로 분리한다. 개발자는 하나의 테이블처럼 쿼리하고, DB 엔진(옵티마이저)이 내부에서 파티션 라우팅을 자동 처리하므로 SQL 변경이 불필요하다. 개념 상세와 4가지 종류(레인지·리스트·해시·컴퍼지트)는 [[파티셔닝]] 참조.

실무 적합 사례:
- **로그 테이블**: 시간 기준으로 지속 누적되는 대용량 테이블 (게임 회사 로그 테이블 실사례로 언급)
- **날짜 기반 최신 데이터 조회**: 조회 조건에 날짜 컬럼이 자주 사용될 때
- **대량 오래된 데이터 일괄 삭제**: 파티션 단위로 DROP해 효율적 정리 가능

### 샤딩 — 다중 인스턴스로의 물리 분산

"옆 동네에 새 집 짓기" — 파티셔닝이 '한 지붕 아래 새 방'이라면, 샤딩은 '아예 새로운 집으로 이사'다. 데이터를 물리적으로 다른 DB 인스턴스(샤드)로 분산·저장하는 수평 확장 전략이다. 샤드 키 기반 라우팅 로직을 **애플리케이션 또는 미들 레이어에서 직접 구현**해야 한다는 점이 파티셔닝과의 결정적 차이다. 개념 상세는 [[샤딩]] 참조.

실무 적합 사례:
- **대규모 트래픽 + 대용량 데이터**: 단일 DB 한계를 넘어설 때
- **글로벌 서비스**: 지역별(한국·미국 등) 데이터 물리 분산으로 데이터 로컬리티 확보
- **스케일업 한계 도달**: 단일 서버 사양을 아무리 높여도 성능 개선 불가할 때

### 핵심 비교표

| 항목 | 파티셔닝 | 샤딩 |
|---|---|---|
| **물리적 위치** | 단일 DB 인스턴스 내부 | 다수의 별도 DB 인스턴스 |
| **확장 방향** | 스케일업(단일 서버 최적화) | 스케일아웃(수평 확장) |
| **관리 주체** | DB 엔진·옵티마이저 자동 처리 | 애플리케이션·미들 레이어 직접 구현 |
| **SQL 변경** | 불필요 | 샤드 키 설계 필수 |
| **아키텍처 레벨** | DB 내부 최적화 기술 | 아키텍처 레벨 분산 전략 |

### 샤딩 도입의 핵심 주의사항

- **조인 연산이 사실상 불가능**해지므로 도입 전 데이터 모델링을 철저히 설계해야 한다.
- 샤딩은 **최후의 수단**에 가까운 전략이다 — 단일 DB 한계를 넘었을 때 비로소 검토한다.
