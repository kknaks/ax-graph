## 0. CRM을 새로 만든다는 것

- 저자: **Frank Koh(고영민)**, Ad Management 파트 서버 엔지니어
- 배경: 버즈빌은 마케팅 리드 확보 → 영업 파이프라인 관리 → 캠페인 운영 → 성과 기반 정산의 긴 업무 흐름을 **Salesforce** 위에서 운영
- 문제: 현업 업무 방식은 계속 바뀌나, 외부 CRM 커스터마이징·내부 시스템 연동 속도가 병목
- 목표: 단순 Salesforce 대체가 아닌 각 담당자의 업무 방식을 전환·개선
- 제약: 무중단 배포, 데이터 마이그레이션, backward-compatibility, 데이터 가시성·정합성 확보 동시 요구

## 1. Salesforce의 비효율

- 버즈빌은 Salesforce를 **10년 이상** 사용, 전담팀 부재·외주사를 통해서만 커스텀 가능
- 사용하지 않는 필드와 규칙이 복잡하게 얽혀 세일즈팀·재무팀 모두 불편에 익숙해진 상태
- 자사 셀프서빙 플랫폼 **광고센터** 출시 후 Salesforce 연동 자동화 → 3개 시스템(광고센터·광고운영시스템·Salesforce) 정합성 유지 난이도 상승
- **atomic 트랜잭션 구현 사실상 불가**, API token 무효화 시 secret rotation·데이터 백필에 긴 시간 소요
- AI Agent 도입에도 Salesforce 비효율이 병목이 되어 전체 업무 효율 개선 미미
- 7월 라이선스 연장을 4개월 앞두고 **연간 라이선스 비용 약 1억 원** 절약 계획 논의 시작

## 2. AX 전환 불확실성 낮추기: ERD & PoC

- 자체 CRM 개발 결정은 되돌리기 어려움 → 이후 유지보수·장애 대응·데이터 정합성 책임 내재화
- 별도 전담 제품팀 없이 디자인·FE 구현을 AI Agent로 완수해야 하는 리소스 제약
- 1차 목표: **ERD**와 **PoC** 제작
  1. **ERD**: 버즈포스 데이터 구조가 Salesforce 주요 데이터를 표현 가능한지, 향후 변경에 유연한지 설명
     - AI Agent를 활용해 엔티티 클릭 시 관계 하이라이트, 업무 시나리오별 관여 엔티티를 보여주는 **인터랙티브 ERD** 제공
     - 막연한 의심 → "세일즈포스의 이 데이터는 왜 버즈포스에서는 이렇게 저장하나요?"라는 구체적 질문으로 전환
  2. **PoC**: 4월 말까지 실제 사용자에게 시연 후 완전 전환 여부 판단

## 3. Salesforce가 버즈빌에서 맡던 역할

- **MC팀**: Pardot(마케팅 툴)으로 캠페인 운영·리드 확보 후 세일즈팀에 전달
- **세일즈팀**: 신규·기존 고객과 미팅 → 광고 집행 제안 → 거래 단위 **"기회(Opportunity)"** 관리
  - 기회: 제안 → 체결 → 광고 운영 → 최종 정산 전체 사이클 담당
- **재무팀**: 광고 집행 종료 기회에 매출·매입 세금계산서 매칭, 광고주 수취·대행사 지급 기록
- **리포트**: 기회·클라이언트 중심 자유 리포트로 영업 현황 파악·의사결정
- 결론: Salesforce = **CRM + 마케팅 + 영업 리포트 + 정산** 통합 시스템

## 4. AI로 가설을 세우고, 인터뷰로 검증

### AI 분석으로 업무 규칙 역추적
- 저자는 Salesforce 직접 사용 경험 없음 → AI Agent로 Salesforce 데이터 패턴 분석
- 예시: 기회가 체결 단계로 넘어가면 집행 시작일·종료일 채움률 **20% → 89%**, 세팅요청 단계에서 Daily Cap·타겟팅 정보 채움률 **90% 이상**
  - 이 분석 → 버즈포스 **Stage 전환 validation** 규칙으로 직결
- 정산 데이터: 일부 대행사는 수십~수백 개 광고 집행 후 통합 세금계산서 1건 발행 → 여러 기회에 분배 매칭 필요
  - 기회별 수수료율 비중 기준 금액 분배 **fallback rule** 도출
  - **`OppBilling`** 별도 테이블 분리의 근거

### 사용자 인터뷰로 유스케이스 검증
- 데이터로 알 수 없는 것: Stage 역방향 이동 여부, 실제 입력 화면 순서, 재무팀 예외 케이스 처리 기준
- 인터뷰 대상 및 질문:
  - **세일즈팀**: 기회가 제안→체결→세팅요청으로 넘어가는 실제 흐름
  - **MC팀**: Pardot 리드가 세일즈팀에 전달되는 기준
  - **재무팀**: 매출·매입 세금계산서·입금 내역 확인 순서, 수동 판단 필요 케이스
- **AI의 역할**: 분석 속도 향상, 가설·인터뷰 질문 목록 생성
- **사람의 역할**: 가설 검증, 제품 반영 규칙 결정, 이해관계자 설득·커뮤니케이션

> "AI는 분석의 속도를 높여주었지만, 그 분석이 실제 업무와 맞는지 확인하고 제품의 책임 범위로 바꾸는 과정에는 커뮤니케이션이 필요했습니다."

## 5. 데이터 재설계: 명시성 중심

### 핵심 원칙
- Salesforce의 하나의 필드가 여러 의미를 가지는 구조 → 데이터 유실 위험 상존
- 버즈포스 데이터 구조: **"명시성"** 중심 설계, 관계 테이블 적극 활용

### 주요 설계 결정

1. **기회-클라이언트 관계 (`OppAccount` 테이블)**
   - 기존 Salesforce: 관련 클라이언트(광고주·대행사·미디어렙사)를 기회 테이블 필드 레벨로 관리 → 3개 이상 클라이언트 참여 시 데이터 유실
   - 버즈포스: `OppAccount` 별도 테이블 분리
     - 하나의 기회 ↔ 여러 클라이언트 연결 가능
     - 각 연결에 `advertiser` / `agency` / `media_rep` **역할**과 수수료율 저장

2. **클라이언트-연락처 관계 (`AccountContact` 중간 테이블)**
   - 실무: 한 담당자가 여러 광고주·대행사와 연결 가능 (N:M)
   - `AccountContact` 중간 테이블로 `role`, `is_primary`, `is_active` 저장
   - 활동 등록·세금계산서 발행 시 휴먼 에러 감소

3. **정산 (`OppBilling` 테이블)**
   - 대행사 통합 세금계산서 1건 → 여러 기회에 금액 분배 매칭 필요
   - 장기 캠페인: 하나의 기회가 여러 번 정산 가능
   - `OppBilling`: Opportunity ↔ Billing **N:M 관계**, 기회별 배정 금액까지 저장

4. **예산 vs 소진액 구분**
   - 기존 Salesforce: 제안 "예산"과 실제 "소진액" 미구분 → 미소진 비율 확인 시 광고운영플랫폼 병행 조회 필요
   - 버즈포스: 명시적으로 분리

## 6. AI 개발의 병목: 피드백 루프

### 배경
- 저자: FE 개발 경험 전무, 버즈포스가 AI Agent를 활용한 첫 FE A-Z 개발 프로젝트
- 기능 구현 자체는 병목이 되지 않음 → **결과물 검증·수정의 피드백 루프**가 병목

### AI Agent의 한계
- "AI가 화면을 빠르게 만든다" ≠ "프론트엔드 엔지니어 역할까지 대체"
- FE 엔지니어라면 스스로 던질 질문("bulk 작업인데 단건 모달만 있으면 안 되지 않나?", "4XX 에러인데 에러 토스트가 불친절하지 않나?")을 Agent는 스스로 던지지 않음
- Agent는 입력된 문제를 빠르게 풀 뿐, **제품 맥락을 지속적으로 붙잡고 자율적 피드백 루프를 돌리지 못함**
- dual write 기간 없이 바로 전환 결정 → 반드시 사용자가 업무 완수 가능한 상태 요구

### 해결 방법
- Agent에게 작업 지시 시 **QA용 테스트 데이터 생성 + QA 가이드라인 HTML 문서** 함께 작성 지시
  - Agent가 staging DB에 접근해 기존 적절한 데이터 선택 또는 신규 생성
  - QA 가이드라인 문서: 여러 기능 동시 개발 중 집중도 유지, 비CLI 환경에서 Agent 지시 인터페이스로 활용
- 출시 후: 여러 Agent를 동시에 돌려 각 이슈를 **병렬 처리**

### 새로운 병목: 맥락 전환 비용
- 여러 Agent 동시 결과물 생성 → 각 작업 배경 재기억·판단·다음 지시 반복
- > "마치 CPU가 실제 연산보다 context switching에 더 많은 시간을 쓰는 상태에 가까웠습니다."
- 정신적 체력 부담도 상당

### 향후 계획
- Agent가 스스로 피드백 루프를 돌릴 수 있는 프레임워크 선행 구축
  - 사용자 시나리오, acceptance criteria, 테스트 데이터를 Agent가 스스로 확인하는 방식
- > "AI 개발의 다음 병목은 코드를 더 빨리 쓰는 것이 아니라 피드백 루프를 얼마나 자율적으로 돌릴 수 있느냐에 있다."

### STICC Framework 활용
- **Situation(상황), Task(작업), Intent(의도), Concern(우려), Calibration(조정)** 5가지 관점으로 작업 지시
- 모호한 프롬프트 대비 작업물 완성도 향상, 미처 생각하지 못한 엣지케이스 발견에도 유용

## 7. 버즈포스가 남긴 것

### 런칭 결과
- **6월 15일** 공식 런칭 후 업무 전환 완료
- 세일즈팀·재무팀 모두 Salesforce 없이 기존 업무 수행 중
- 7월 초 기준 6월 집행 광고 정산 업무도 버즈포스에서 진행
- 리포트 고도화·광고운영시스템 연동·기능 개선·버그 픽스 계속 진행 중

### 사용자 반응 차이
- **세일즈팀**: "기존 업무를 문제 없이 이어갈 수 있다" 수준
  - 버즈포스 = Salesforce 밖에서 일어난 일을 기록·조회하는 시스템
  - 엔티티·데이터 중심 사용
- **재무팀**: "업무가 많이 효율화되었다" 명확한 반응
  - 매입 세금계산서-기회 매칭, 매출 세금계산서 업로드 등 **행동·유스케이스 중심** 사용
  - 기존 Salesforce: 유스케이스가 기회 상세 페이지 곳곳에 흩어져 있어 불편
  - 버즈포스: 유스케이스 단위로 페이지·URL 분리 + 자동화 기능 추가

### 핵심 교훈
- 업무 시스템 만족도는 **"사용자의 일을 어떤 단위로 제품화했는가"**에 크게 좌우
- 플랫폼 안에서 많은 일 처리 가능 + 플랫폼 밖 업무와 자연스럽게 연결 + 유스케이스 단위로 이해·실행 가능할 때 변화 체감 극대화
- 비개발자도 자신의 업무를 유스케이스 단위로 정의할 수 있어야 Agent 기반 업무 효율화가 쉬워짐
- **"모든 것을 의심하는 태도"**가 AX 시대의 중요한 덕목
- 출발점: "당연해서 의심하지 않았던 것이 뭘까?"
- AI와의 분업: AI = 빠른 구현·반복 작업·context switching 보조 / 사람 = 도메인 지식 전달·사용자 니즈 파악·설득·프로젝트 주도

### 향후 계획 (하반기)
- 리소스의 20~30% → 영업·운영 효율화
- 광고운영시스템과 버즈포스 연동 강화 (운영자 직접 개입 프로세스 축소)
- 버즈빌 영업 병목 파악·분석 인터페이스 마련
- 버즈포스 데이터를 분석계에 연동 → LLM 기반 분석 지원
