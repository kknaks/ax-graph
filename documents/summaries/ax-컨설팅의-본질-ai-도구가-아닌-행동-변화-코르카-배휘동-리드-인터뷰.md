## 배휘동 소개 및 커리어 여정

- 실리콘밸리 AI 번역 스타트업 **엑셀레이(Accelera)**에서 약 3년 반간 프론트엔드 팀 리드로 근무
- 바이브 코딩·AI 분야에서 의도치 않게 주목받으며 교육·강의·코칭·컨설팅 활동 시작
- 현재 **코르카(Korca)** AX팀 리드로 재직 중
- 코르카는 원래 B2C 프로덕트(AI 논문 도우미 **문라이트**, AI 캘린더 **트레이스**) 회사였으며, 배휘동 합류 후 AX 컨설팅으로 사업 확장
- 문라이트는 글로벌 사용자 약 13만 명, 딥시크 이슈로 중국 VPN 유입도 발생

## AX란 무엇인가

- **AI 트랜스포메이션(AX)**: AI를 통해 개인·팀·조직의 행동을 함께 바꾸는 작업
- 대부분 조직은 '생산성 향상'에 초점을 맞추지만, 코르카는 생산성 향상을 위해 **개인·팀·조직이 동시에 변해야 한다**는 관점을 취함
- 도구만 제공하고 끝나는 방식으로는 실질적 행동 변화·지표 변화가 일어나지 않음
- 도구를 준 뒤 결과 추적 없이 "알아서 하시고"로 끝나는 패턴이 일반적이며 이를 문제로 봄

## AX 컨설턴트 역할과 채용 기준

세 가지 핵심 역량:
1. **문제를 발견하는 역량** — 하루·일주일 작업 로그를 통해 병목과 비효율을 찾음
2. **해결책을 E2E로 직접 구현하는 역량** — SaaS 스타일 솔루션, 프로덕트 실험까지 직접 완성
3. **변화를 만들어 내는 역량** — 구현 후 실제 조직 행동이 바뀌고 임팩트가 생기도록 끌고 가는 것

- 채용 프로세스: 이력서에서 세 역량이 드러나야 하며, 유망 지원자는 **2~3시간 짝 작업(pair session)** 진행
- 채용 공고를 올렸을 때 지원자보다 협업 제안이 더 많이 들어옴

## 프론트엔드 개발자에서 AX 컨설턴트로의 전환

- 전산과 출신으로 동아리 홈페이지 제작 계기로 개발 시작 (jQuery, PHP 독학)
- 프론트엔드가 맞았던 이유: ① 오타·UI 오류 감지 능력, ② 다양한 서비스 레이어 통합 역량
- 애자일 커뮤니티 **AC2**(김창준 님 운영)를 2011~2012년부터 참여하며 코칭·멘토링·컨설팅에 관심 형성
- 엑셀레이 구조조정 시 자발적 퇴직 의사 → 파트타임(반일·반급여)으로 글쓰기·컨설팅 시작
- 바이브 코딩 붐과 함께 강의 요청 급증 → 회사 병행 어려워 독립 후 코르카 합류

## AX 프로젝트 접근법: 문제 발견 방법론

- 클라이언트에게 요청하는 첫 과제:
  1. 지난 하루를 돌아보며 시간·에너지를 어디에 얼마나 썼는지 텍스트로 기록
  2. 일주일치 데이터로 확장
  3. AI 보조 프롬프트로 분석 지원
- 고통스럽거나 남의 도움이 필요한 작업을 선정 → 현재 처리 방식을 기술
- 비개발자의 경우 AI·프로그램의 가능/불가 영역 구분을 도와주는 역할 수행
- **효능감 우선 전략**: 일부라도 자동화·프로그램화 → 성공 경험 → 점진적 확장
- 상황에 따라 직접 구현, 프롬프트 제공, 반나절 공동 작업 등 형태를 다르게 적용

## 집체 교육의 한계와 챔피언 전략

- 집체 교육의 문제:
  - 도구의 '스냅'만 가져오고 그 도구가 탄생한 내부 히스토리·맥락은 전달되지 않음
  - 규칙으로만 작동 → 구성원의 불만, 효능감 부재
  - 업무 중 추가 교육에 대한 피로감 ("바빠 죽겠는데 또 하라고")
  - 평소 행동 양식과 차이가 크면 인지 부하가 높아 채택되지 않음
- **챔피언 전략**:
  - 조직 내 의지 높은 개인을 먼저 변화시켜 **체인지 에이전트**로 만듦
  - 동료가 변화하는 모습을 보며 FOMO(조급함)가 생기고, 자발적 동기 유발
  - **탑다운**(경영진 지원·도구 제공)과 **바텀업**(동료 성과 목격)의 동시 접근 필요
  - 정기 세션으로 내부 공유 지속 → AX 문화 확산
- 강화학습의 **온 폴리시(on-policy)** 개념 유추: 외부 사례를 그대로 이식하는 것(오프 폴리시)보다, 조직 구성원 자신의 행동에서 시작하는 델타 변화가 진짜 내재화로 이어짐 (배휘동 본인의 해석으로, 정확성 유보)

## 문라이트 코드 품질 개선 사례

### 배경
- 문라이트 레거시 코드베이스에 배휘동과 시니어 엔지니어 **강규형** 님이 합류
- 목표: 팀 방해 최소화, 점진적 코드 품질 향상, AI와 사람 모두 더 잘 일할 수 있는 환경 조성

### 코드 품질과 AI의 관계
> "코드베이스 품질이 높아질수록 AI도 훨씬 더 일을 잘할 수 있다."
- 정렬된 임포트 순서 등 일관된 코드 스타일 → AI가 유사한 고품질 코드 패턴을 참조할 확률 증가
- AI의 실수 확률 감소, 개발 속도·품질 동시 향상

### 타이딩(Tidying) 도입
- **타이딩**: 켄트 벡(Kent Beck)이 정의한 안전한 소규모 리팩토링
  - 예: 코드 뭉치 사이 빈 줄 추가, 코드 위치 이동 등 의미 변경 없는 정리
- 초기 문제: 팀원들이 타이딩을 잘 안 함
  - 원인 ①: 코드베이스에 너무 익숙해져 무엇을 타이딩해야 할지 모름
  - 원인 ②: 변화가 너무 작아 임팩트가 없다고 느낌
- 해결책:
  1. 타이딩 대상을 제안해 주는 **프롬프트 문서** 작성 (레스 모어 원칙으로 압축)
  2. 단순 코드가 아닌 **최근 커밋**을 분석 대상으로 설정 → 관심도·임팩트 향상

### 서브 에이전트 자동화 워크플로우
- **Claude Code 커맨드**로 프롬프트 등록
- **타이디 타겟 커밋(tidy-target-commit)** 셸 스크립트: git log에서 타이딩 대상 커밋 필터링
- 병렬 서브 에이전트로 각 커밋을 동시에 분석 → 타이딩 제안 목록 생성
- 오케스트레이션 에이전트가 제안을 취합 → 사람(개발자)이 검토 후 승인·적용
- LLM이 할 수 있는 것과 스크립트로 할 수 있는 것을 분리해 토큰 비용·속도 최적화
- 장기 목표: 충분한 코드 품질 확보 후 매일 수십 개 커밋을 자동 타이딩하는 완전 자동화

### 코드 품질 지표 관리
네 가지 지표(자료에서 확인된 세 가지):
- **린트 오류 수**: 수백 개에서 수십 개로 대폭 감소 (ESLint 규칙 정리 + 일괄 픽스)
- **Knip(립)**: 죽은 코드(미참조 코드) 제거 추적
- **오류 밀도**: (린트 오류 + Knip 오류 + α) ÷ 라인 수 → 코드 증가에도 오류 비율 하락을 측정
- 지표는 계단식으로 개선 — 소규모 타이딩 축적 후 대담한 결정을 내릴 수 있는 시점에 큰 폭 개선
- 문라이트 팀은 개발자·비개발자 구분 없이 전 팀원이 PR을 올리는 AI 네이티브 문화

## 짝프롬프팅(Pair Prompting)

- 배휘동이 올해 중순부터 주당 5~6시간씩 진행한 공동 작업 방식
- **짝프롬프팅**: 하나의 태스크에 대해 각자 프롬프트를 작성 → 비교·취합 → 각자 실행 후 결과 비교
  - 실행 전 "이 프롬프트로 어떤 결과가 나올 것 같은가?" 예측·토론
  - 키워드 선택, 컨텍스트 구성 등 세부 차이를 통해 AI 활용 역량 압축 성장
- 파트너: 강규형 님, 수미 님 등

## AX 미래 전망

- AX팀·TF 조직 형태가 얼마나 오래 갈지는 불확실
- 구글 워크스페이스 등 벤더가 기능을 내재화할 경우 별도 AX팀은 흡수될 수 있음
- 그러나 **"변화를 만들고 일을 더 잘하고자 하는 니즈 자체는 계속 존재"**
- DX(디지털 트랜스포메이션)가 오래됐음에도 AX 컨설팅 현장에 가보면 DX조차 완성되지 않은 경우가 많음 → 다음 레벨도 유사한 패턴 반복 예상
- AX팀이 사라져도 그 본질적 업무는 살아남을 것

## 시니어를 위한 조언

> "지금 여러분이 손으로 하고 있는 그 작업은 언제나 프로그램화될 수 있다고 생각해 보고 한번 만들어 보시라."

- 실천 예시: 배휘동이 해커톤 심사를 위해 구글 앱스 스크립트로 자동화
  - 구글 시트 데이터 → 구글 독스(이미지 포함) 자동 생성 → 공유 링크를 시트에 삽입 → 상위 후보가 포함된 구글 폼 생성
  - 과거에는 손으로 했을 작업을 약 1시간 만에 자동화
- 핵심 태도: **호기심** + 직접 도전해 효능감 확인
