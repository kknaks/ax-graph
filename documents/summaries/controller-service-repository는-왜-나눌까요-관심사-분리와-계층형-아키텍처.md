## 도입 — 식당 비유

- **식당 비유**: 주문받는 사람(홀) / 요리하는 사람(주방) / 재료 관리하는 사람(창고)이 각각 따로 존재
- 한 사람이 모든 일을 하면 주문이 밀리는 순간 주방도 창고도 멈춤
- 소프트웨어도 동일: 코드도 각자 한 가지 일만 맡도록 영역을 나눠야 함 → **관심사 분리**와 **계층화**

---

## 관심사 분리(Separation of Concerns) 개념

- 정의: 서로 다른 종류의 일을 각각 다른 영역으로 나누는 설계 원칙
- 핵심 문장: **"한 코드 조각은 한 가지 관심사만 책임진다"** — 그래야 바뀌는 이유도 하나가 됨

### 대표적인 관심사 세 가지

| 관심사 | 역할 |
|--------|------|
| **표현** | 요청을 받고 응답 형식을 만드는 일 |
| **비즈니스** | 핵심 규칙과 흐름을 처리하는 일 |
| **영속** | 데이터를 저장하고 조회하는 일 |

- 세 가지는 성격이 완전히 다르기 때문에 섞어 두면 안 됨

### 뒤섞였을 때 문제

- 할인 정책 하나만 바꾸려 해도 요청 처리·저장 코드까지 들어 있는 거대한 메서드를 건드려야 함
- 비즈니스 로직만 따로 테스트하고 싶어도 웹 요청·데이터베이스가 묶여 있어 테스트 어려움
- 같은 로직을 다른 곳에서 재사용 불가

## 계층형 아키텍처 기본 구조

- 클라이언트 요청 → **Controller** → **Service** → **Repository** → **Database**
- 응답은 역순으로 거슬러 올라옴
- **핵심 규칙**: 각 계층은 바로 아래 계층만 호출하고, **계층을 건너뛰지 않는다**
  - 컨트롤러가 서비스를 건너뛰고 리포지토리를 직접 호출하면 안 됨
  - 이 규칙이 지켜져야 각 계층이 독립적으로 동작하고 중간 계층을 교체 가능

### 각 계층의 역할

| 계층 | 해야 할 일 | 하지 말아야 할 일 |
|------|-----------|------------------|
| **Controller** | 요청·응답 변환, 입력 검증 | 비즈니스 규칙 판단 |
| **Service** | 비즈니스 규칙, 트랜잭션 경계 | HTTP·SQL 직접 처리 |
| **Repository** | 데이터 저장·조회 | 비즈니스 판단 |

- 코드를 어느 계층에 둘지 판단 기준: **"이 코드가 바뀌는 이유가 무엇인가?"**
  - 화면 형식 때문 → Controller
  - 정책 때문 → Service
  - 저장 방식 때문 → Repository

## 코드 예시 비교

### 나쁜 예 — 뒤섞인 컨트롤러

- 주문 생성 메서드 하나 안에 세 가지 일이 혼재:
  1. 금액이 음수인지 검사하는 **입력 검증**
  2. 만 원 초과 시 1,000원 할인하는 **비즈니스 규칙**
  3. 데이터베이스에 직접 INSERT문 실행하는 **저장**
- 요청을 Map으로 받아 타입 안정성 없음
- 바뀌는 이유가 너무 많음: 할인 정책·저장 방식·요청 형식 변경 모두 이 컨트롤러를 수정해야 함

### 좋은 예 — 계층 분리

**Controller**
- `@Valid` 어노테이션이 붙은 `OrderRequest`로 요청 수신
- Command 객체로 변환해 Service에 위임
- 결과를 응답 객체로 변환해 반환
- 입력 검증은 `@Valid`가 처리, 할인 규칙·저장은 Service·Repository로 위임
- 역할: **통역사** — 들어온 요청을 내부 형태로 바꾸고, 나온 결과를 외부 형식으로 바꿈
- 할인 정책이 바뀌어도 Controller는 수정 불필요

**Service**
- `@Transactional` 어노테이션으로 **트랜잭션 경계** 설정 — 메서드 안 작업이 전부 성공하거나 전부 취소
- `Order.create()` 호출로 주문 객체 생성 → **핵심 규칙을 도메인 객체 안에 위치**
- Service는 비즈니스 흐름 조율에 집중, 실제 규칙은 도메인이 담당 → Service가 얇게 유지됨

**Repository**
- 데이터 저장소 접근을 **추상화**하는 계층
- Service는 데이터가 어디에 어떻게 저장되는지 몰라도 됨 — `save()` 인터페이스(명세)만 알면 충분
- 추상화의 세 가지 이점:
  1. 저장 기술이 바뀌어도 Service 코드 그대로
  2. 테스트 시 가짜 Repository 구현을 끼워넣어 Service만 단독 테스트 가능
  3. 쿼리를 바꿔도 비즈니스 로직에 영향 없음
- Spring Data JPA: 인터페이스만 선언해도 구현체를 자동 생성 → 추상화를 거의 공짜로 획득

## DTO 경계

- 데이터 흐름: 클라이언트 요청 → **요청 DTO** → (내부) **도메인 객체** → **응답 DTO** → 클라이언트
- **DTO**: 데이터를 실어나르는 전용 객체
- 엔티티를 외부에 직접 노출하면 두 가지 문제:
  1. 내부 데이터 구조가 그대로 노출됨
  2. 비밀번호 같은 민감한 필드까지 응답에 포함되는 **보안 사고** 가능성
- DTO는 계층 사이를 막아주는 **방어망** 역할

## 의존성 방향 규칙

세 가지 원칙:

1. **단방향**: 위 계층이 아래 계층에 의존, 역방향은 절대 금지
2. **계층 건너뛰기 금지**: Controller가 Service를 건너뛰고 Repository를 직접 호출 불가
3. **추상화 의존**: 구체적인 구현이 아니라 인터페이스에 의존

- 가장 중요한 원칙은 **단방향**
- Service가 Controller를 알아서는 안 됨: 비즈니스 로직이 특정 표현 방식(웹 화면)에 묶이면 배치나 다른 진입점에서 재사용 불가
- 의존 방향이 한쪽으로만 흐를 때 비로소 각 계층을 독립적으로 교체 가능

## 계층화의 트레이드오프

| 장점 | 비용 |
|------|------|
| 변경이 한 계층에 격리됨 | 코드량 증가 |
| 테스트 용이 | 단순 기능도 여러 파일 필요 |
| 역할 명확 | 단순 CRUD에 기계적 적용 시 보일러플레이트만 증가 |

- **실용적 기준**: 변경과 테스트가 필요할 만큼 복잡도가 생길 때 분리
- 모든 코드를 똑같이 나눌 필요는 없음

## 흔한 안티패턴

| 안티패턴 | 설명 | 해결책 |
|----------|------|--------|
| **뚱뚱한 컨트롤러** | 컨트롤러에 비즈니스 로직이 들어감 | Service로 이동 |
| **빈약한 서비스** | Service가 단순 호출만 전달 | 도메인 객체에 규칙 부여 |
| **엔티티 노출** | 엔티티를 응답으로 직접 반환 | 응답 DTO로 변환 |
| **계층 침투** | Service 안에 HTTP 요청 객체 등장 | 계층 전용 객체 사용 |
| **양방향 의존** | 아래 계층이 위 계층을 참조 | 단방향으로 정리 |

- 공통 원인: **한 계층이 자기 책임을 넘어서는 일을 함**

## 핵심 요약 및 실무 적용 팁

**핵심 요약**
- 관심사 분리: 다른 종류의 일을 다른 영역으로 나누는 원칙
- Controller: 요청·응답 변환·검증 / Service: 비즈니스 흐름·트랜잭션 경계 / Repository: 데이터 접근 추상화
- 계층 사이에 DTO를 두어 엔티티 노출 방지
- 의존 방향은 위→아래 단방향, 계층 건너뛰기 금지

**실무 적용 팁 4가지**
1. 코드를 짤 때 "이게 바뀌는 이유가 무엇인가"를 기준으로 계층 결정
2. 엔티티는 계층 밖으로 내보내지 말고 DTO로 변환해 반환
3. Service가 얇으면 도메인 객체에 규칙을 넣어 빈약한 서비스 방지
4. 단순 CRUD까지 기계적으로 나누지 말고 복잡도가 생길 때 분리
