## 도입: 프롬프트 시대의 종말

- 한 엔지니어가 Claude Code에 한 문장을 입력하고 커피를 가지러 갔다 돌아오니 **에이전트 40개**가 코드를 작성·검토하고 **풀 리퀘스트 3개**를 남겨놨다.
- 그가 쓴 것은 프롬프트 40개가 아니라 **루프 하나**였다.
- **Claude Code 책임자(Anthropic)**의 발언:
  > "I do not prompt Claude anymore. I have loops running that prompt Claude and figure out what to do. My job is to write loops."
- 2026년 6월 한 주 만에 **'루프 엔지니어링(Loop Engineering)'**이라는 이름과 12페이지 분량의 현장 가이드가 등장해 확산됐다.

---

## 루프 엔지니어링이란?

- **기존 방식**: 프롬프트 입력 → 결과 확인 → 다음 프롬프트 입력 — 개발자가 루프 그 자체였다.
- **루프 엔지니어링**: 개발자가 내부 사이클에서 한 단계 올라가, 에이전트가 달리는 **트랙(시스템)을 설계**한다.
- 루프의 4단계 반복 구조:
  1. Discover(작업 발견)
  2. Plan(계획)
  3. Execute(실행)
  4. Verify(검증) → 조건 충족 시 종료, 아니면 반복

### AI 작업 패러다임의 연대기
| 연도 | 역할 | 행위 |
|------|------|------|
| 2024 | 운영자(Operator) | 프롬프트 작성 |
| 2025 | 관리자(Manager) | 여러 에이전트 동시 운영 |
| 2026 | 시스템 설계자(System Designer) | 에이전트를 구동하는 시스템 설계 |

### 스택 구조
- **프롬프트 → 컨텍스트 → 하네스(harness)** — 단일 에이전트 실행에 필요한 모든 것
- **루프 엔지니어링** — 하네스를 타이머 위에서 구동하는 외부 시스템, 스택의 한 층 위

## 루프의 5가지 핵심 요소 + 메모리

### 1. 자동화(Automations) — 심장박동
- 스케줄에 따라 자동으로 작업을 **발견·분류(triage)**하고 인박스에 적재
- 이것이 루프를 '한 번의 실행'이 아닌 **진짜 루프**로 만드는 요소

### 2. 워크트리(Worktrees)
- 에이전트 2개 이상을 동시에 실행하면 파일 충돌 발생
- **Git 워크트리**가 각 에이전트에 독립 체크아웃·독립 브랜치를 제공 → 서로의 작업에 간섭 불가

### 3. 스킬(Skills)
- 프로젝트 지식(컨벤션, 빌드 절차, 특수 규칙 등)을 **한 번 문서화**한 것
- 없으면 루프가 매 사이클마다 프로젝트 전체를 처음부터 재파생 → 비효율

### 4. 커넥터(Connectors) — MCP 기반
- **MCP(Model Context Protocol)**로 실제 도구(이슈 트래커, DB, Slack, Stripe 등)에 연결
- "여기 수정 방법입니다"라고 말하는 에이전트 vs. **풀 리퀘스트를 직접 여는 루프**의 차이

### 5. 서브에이전트(Sub-agents)
- 루프에서 가장 유용한 단일 이동: **작성자와 검토자를 분리**
- 코드를 쓴 모델은 자신의 코드를 채점할 때 너무 관대함 → 다른 지시를 받은 **별도 에이전트**가 첫 번째가 납득시킨 오류를 포착

### 메모리(Memory) — 척추
- 마크다운 파일, Linear 보드 등 **대화 외부에 존재**하는 것
- 에이전트는 실행 간 모든 것을 잊지만, **레포는 잊지 않는다**
- 완료된 것·다음 할 것을 기록하는 메모리가 루프 전체의 연속성을 유지

## 제너레이터 vs. 버파이어: 진짜 병목

- **루프 = 제너레이터(Generator) + 버파이어(Verifier)**
  - 제너레이터: 작업을 생산하는 모델 — 현재 모델들은 이미 매우 뛰어남
  - 버파이어: 작업이 충분히 좋은지 판단하는 것 — **진짜 병목**
- 약한 버파이어의 결과:
  > 랜딩 페이지를 더 좋게 만들어. 계속해.
  - '더 좋다'의 정의 없음 → 히어로 섹션을 8번 다르게 재작성, 어느 것도 명확히 더 낫지 않음, 성공 보고
  - 실제 돈을 지불하고 **진전 없는 움직임**만 얻는 **슬롭 머신(slop machine)**

### 버파이어 = 보상 함수 설계
- 버파이어를 작성하는 것은 **보상 함수(reward function)를 정의**하는 것
- 모델을 훈련하는 게 아니라 **'좋다'의 의미를 정의**하는 것
- 개발자의 취향·판단·올바름에 대한 감각 → **해자(moat)**
- 모델은 범용 상품(commodity) → **에이전트가 자신의 작업을 스스로 채점하게 두면 안 됨** (항상 A학점을 줌)
- **Claude Code의 `goal` 명령**: 종료 결정을 별도의 빠른 모델에 위임 — 코드를 쓴 모델이 완료를 선언하지 않음

## 닫힌 루프 vs. 열린 루프

| 구분 | 닫힌 루프(Closed-Loop) | 열린 루프(Open-Loop) |
|------|------------------------|----------------------|
| 특징 | 성공 조건을 사전에 고정, 매 단계 검사, 명령 시 정지 | 목표와 탐색 여지만 부여 |
| 장점 | 예측 가능, 저비용, 방치 안전 | 새로운 해결책 발견 가능 |
| 단점 | 창의적 탐색 제한 | 토큰 소모 많음, 슬롭 위험 |

- **열린 루프 구제 트릭**: 하드 체크를 바닥으로 유지하면서 열린 지시 하나 추가
  > "헤드라인은 놀라게 해줘"
  - 탐색은 허용하되 기준 이하로 내려갈 수 없음
- **버파이어**가 두 종류 모두를 실제로 배포 가능하게 만드는 핵심

## 자율성의 사다리

- 인프라를 처음부터 만들 필요 없음. 명확한 단계:
  1. 베어 셸 루프 — 개발자가 정지 버튼
  2. `goal` 명령 — 소형 모델이 '완료' 판단
  3. 목표 추적 설정
  4. 항상 켜진(never-sleep) 에이전트
- **원칙: 작업을 수행하는 가장 덜 자율적인 도구를 선택하라**

---

## 루프의 복리 효과

- 루프 A: 30분마다 지원 티켓 처리 + 학습 내용을 공유 폴더에 기록
- 루프 B: 콘텐츠 작성
- 루프 C: 로드맵 우선순위 결정
- 모두 **같은 파일을 읽고 씀** → 한 루프가 배운 것을 다른 루프가 모두 흡수
- 한 개발자가 이 시스템으로 **하루 20~40페이지를 직접 감시 없이 배포**

---

## 에이전트 친화적 코드베이스 조건

루프가 작동하려면 코드베이스가 준비되어 있어야 함:
- **가독성(Legible)**: 에이전트가 무엇을 변경할지 찾을 수 있도록
- **실행 가능성(Executable)**: 개발 서버가 이미 켜져 있어야
- **검증 가능성(Verifiable)**: 작업의 증거를 제시할 수 있어야 (예: 브라우저 테스트가 클립을 녹화)

---

## 루프가 해결하지 못하는 세 가지 문제

1. **검증은 여전히 개발자의 몫** — '완료'는 주장일 뿐, 증명이 아님
2. **이해력 부식** — 루프가 만든 것을 읽기를 멈추면 이해력이 썩음
3. **편안한 함정** — 루프가 주는 것을 그냥 받아들이는 것이 위험한 선택

## 최종 판정: 유행어인가, 실질적 전환인가?

- 용어 자체는 주목을 끌기 위해 만들어졌고, 에이전트 빌더들이 이미 알던 내용을 일부 재포장한 것
- **그러나 전환 자체는 실재함**:
  - 에이전트를 직접 프롬프트하는 대신 에이전트를 구동하는 시스템을 설계하고 있는가? → **그렇다**
  - 시스템 정의가 희소한 스킬인가? → **그렇다**
- **결정적 차이**:
  - 루프를 자신이 깊이 이해하는 작업에 더 빠르게 활용하는 사람
  - 루프를 이용해 작업 이해 자체를 회피하는 사람
  - 루프는 그 차이를 모른다. **개발자는 안다.**

## 핵심 요약

> **에이전트에 프롬프트하지 마라. 에이전트를 프롬프트하는 루프를 설계하라.**
> 5가지 요소 + 메모리. 제너레이터와 버파이어의 결합, 버파이어가 전부. 루프를 한 번 만들고 엔지니어로 남아라.
