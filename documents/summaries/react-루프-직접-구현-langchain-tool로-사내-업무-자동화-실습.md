## 핵심 개념 — LLM과 도구의 관계

- **LLM = 두뇌**: 추론·판단을 담당
- **도구(Tool) = 손발**: LLM이 직접 수행할 수 없는 외부 기능(검색, DB 조회, API 호출 등)에 연결하는 실행 수단
- LLM 단독으로는 실시간 정보 조회·사내 시스템 연동·계산 등이 불가능하므로, 도구를 통해 능력을 확장

## @tool 데코레이터 — 개념과 동작 원리

### 역할
- 일반 Python 함수를 LangChain 도구 객체로 변환
- 가장 단순하면서도 강력하고, 실무에서 가장 많이 사용되는 도구 정의 방법

### 모델에 전달되는 정보 vs 전달되지 않는 정보

| 구분 | 내용 |
|---|---|
| **전달됨** | 함수 이름, **docstring**(도구 용도 설명), **타입 힌트**(인자 스키마) |
| **전달 안 됨** | 함수 내부 실행 로직(실제 비즈니스 코드) |

- 모델이 도구를 호출할지, 어떤 인자로 호출할지는 **함수 정의부(이름·docstring·타입힌트)** 에만 의존
- 따라서 docstring에 도구 용도·입력 형식(날짜 포맷 등)을 충분히 기재하는 것이 중요

### 도구 스키마 구성 요소
- `name`: 함수 이름
- `description`: docstring 내용
- `args_schema`: 타입 힌트 기반 인자 스키마 (JSON Schema 형태로 자동 생성)

### 실행 방법
- 코드에서 직접 실행: `.invoke({"인자명": 값})` — 다른 LCEL 컴포넌트와 동일 인터페이스

## bind_tools — 모델에 도구 등록

### 역할
- 정의된 도구들의 **스키마 정보**를 모델에 전달하는 메소드
- 모델이 응답 시 어떤 도구를 어떤 인자로 호출할 수 있는지 파악할 수 있게 함
- 실제 도구 실행은 bind_tools가 하지 않음 — 모델이 "어떤 도구를 호출하겠다"는 의사를 `tool_calls` 필드로 표현할 뿐

### 동작 흐름
1. `model.bind_tools(tools)` → 도구 스키마가 모델 컨텍스트에 포함됨
2. `model.invoke(messages)` → 모델 응답에 `tool_calls` 포함 가능
3. `tool_calls`가 있으면 외부 코드(루프)가 실제 도구를 실행하고 결과를 ToolMessage로 반환

## ReAct 패턴 — 이론

### 개념
- **ReAct** = **Re**asoning(추론) + **Act**(행동) + **Obs**erve(관찰)의 반복 사이클
- LLM 에이전트가 복잡한 작업을 처리할 때 사용하는 핵심 아키텍처 패턴

### 사이클 상세

1. **Reasoning(추론)**: 사용자 질문을 받아 현재 보유한 정보로 답변 가능한지 판단. 필요한 도구가 있는지 결정
2. **Act(행동)**: 필요한 도구를 선택하고 `tool_calls`로 호출 의사 표현 → 외부 코드가 실제 실행
3. **Observe(관찰)**: 도구 실행 결과(ToolMessage)를 메시지 히스토리에 추가 → 모델이 결과를 확인
4. 위 사이클 반복 — 충분한 정보가 수집되면 도구 호출 없이 **최종 텍스트 응답** 출력

### 종료 조건
- 모델 응답의 `tool_calls`가 비어 있을 때 → 루프 종료, 최종 답변 반환
- `content` 필드에 텍스트가 있고 `tool_calls`가 없는 상태가 최종 응답

> 강사(판다스 스튜디오): "리즈닝(추론)하고 액트(행동)하고 — 행동은 도구를 호출하는 과정 — 옵저브는 도구를 실행한 결과 호출 결과를 확인하는 것. 이 사이클을 ReAct 패턴이라고 이야기한다."

## 사내 업무 도구 5종 구현

모두 실제 시스템 연동 없이 **하드코딩 시뮬레이션 데이터**로 구현 (실습 목적).

1. **일정 조회 (schedule)** — 날짜 문자열 입력 → 해당 날짜 일정 리스트 반환
   - 시뮬레이션 데이터: 4월 15일 일정 2개, 4월 16일 일정 1개
2. **휴가일수 계산** — 시작일·종료일 입력 → **주말 제외 영업일 기준** 휴가일수 반환
3. **연차 조회** — 사원번호 입력 → 남은 연차일수 반환 (3명 데이터 하드코딩)
4. **Slack 공지 발송 (send_notification)** — 채널(`team-dev`, `announcements`)·메시지 입력 → 로그 출력 후 실행 결과 메시지 반환 (실제 Slack API 미구현)
5. **IT 헬프데스크 검색** — 키워드 입력 → 관련 작업 내용 반환 (하드코딩)

## 외부 API 도구 — Tavily 웹검색

- Tavily API 키를 `.env`에 설정 후 초기화
- 설정: 최대 3개 결과, 일반 검색 모드
- **폴백 패턴**: API 키 없을 시 하드코딩 목업 데이터 반환
- 검색 결과: URL + 웹페이지 본문 내용 포함
- 사내 도구 5종과 결합해 모델에 함께 바인딩

## 수동 ReAct 루프 구현 (~30줄)

### 사전 준비
- 도구 맵핑 딕셔너리 생성: `{"도구이름": 도구객체}` — 이름으로 도구를 빠르게 조회하기 위함

### 루프 함수 구조 (의사코드)

1. 메시지 리스트 초기화 (사용자 질문 포함)
2. **루프 시작**
   - `model_with_tools.invoke(messages)` 실행 → AIMessage 반환
   - AIMessage를 메시지 리스트에 추가
   - `response.tool_calls`가 비어 있으면 → **루프 종료**, `response.content` 출력
   - `tool_calls`가 있으면:
     - 각 tool_call에서 `name`(도구 이름)·`args`(인자)·`id`(tool_call_id) 추출
     - 로그 출력 (어떤 도구를 어떤 인자로 호출하는지)
     - 맵핑 딕셔너리에서 도구 객체 가져와 `.invoke(args)` 실행
     - 실행 결과를 **ToolMessage**로 감싸 메시지 리스트에 추가
     - ⚠️ **`tool_call_id` 반드시 매핑**: 에이전트가 어떤 도구 호출의 결과인지 식별하는 유일한 키
3. 루프 반복

### tool_call_id 매핑의 중요성
- 모델은 여러 도구를 동시에 또는 순차적으로 호출할 수 있음
- ToolMessage에 `tool_call_id`가 없거나 잘못 매핑되면 모델이 어떤 호출의 결과인지 파악 불가
- 결과: 잘못된 추론 또는 오류 발생

## 테스트 시나리오

### 시나리오 1 — 단일 도구 호출
- **질문**: "4월 15일 일정 알려줘"
- **동작**: `schedule` 도구 1회 호출 → 일정 2건 확인 → 최종 답변 생성
- **결과**: 정상 동작 확인

### 시나리오 2 — 다중 도구 호출 (복합 질문)
- **질문**: 특정 직원(사원번호 지정)이 4월 21~24일 휴가 시 남은 연차일수 계산
- **이상적 도구 호출 순서**: ① 연차 조회 → ② 영업일수 계산 → ③ (공휴일 포함 여부 확인)
- **Q1 32B 모델 실제 결과**: ① 연차 조회 + ② 영업일수 계산 2회만 호출 (③ 누락)
- **강사 코멘트**:
  > "상용 모델인 OpenAI나 Gemini 이런 모델들 한번 사용해보시면 결과가 달라질 수 있으니까 직접 실습하시면서 그 차이도 확인해 보시면 좋겠습니다."
- **시사점**: 동일한 ReAct 루프라도 **모델 성능에 따라 도구 호출 횟수·정확도가 달라짐**

### 시나리오 3 — 이종 도구 순차 호출
- **질문**: LangChain 최신 변경점을 웹에서 검색한 뒤 Slack team-dev 채널에 공지
- **동작**: ① Tavily 검색 → ② 결과 요약 → ③ `send_notification`으로 `team-dev` 채널 발송
- **결과**: 업무 성격이 다른 두 도구를 순차적으로 정확히 처리 확인
