## 1. 봇(Bot)이란 무엇인가

> "반복적인 작업을 사람 대신에 수행하는 소프트웨어 프로그램"

- **봇 예시**
  - **GitHub Actions**: 배포 자동화 (없으면 EC2 직접 접속해서 수동 배포)
  - **Jira Automation**: PR 연결 시 티켓 자동 업데이트
  - **챗봇(CS 툴)**: 사람 대신 고객 응대 자동화
  - 공연 티켓 매크로: 안 좋은 사례 — 사람보다 빠른 자동 선점

---

## 2. 슬랙봇이란

- 발표자 정의: **슬랙과 인터랙션하는 봇**
  - 외부 이벤트 → 슬랙으로 알림 수신
  - 슬랙 명령 → 외부 도구에 직접 명령 수행
- 봇 자체와 슬랙은 별개 — 봇의 입출력 인터페이스가 슬랙인 것
- **슬랙봇 예시**
  - **Polly**: 회식 날짜 등 설문 봇
  - **Donut**: 랜덤 그룹 미팅 주선 봇
  - **Jira 슬랙봇**: 알림 수신 + 슬랙 내에서 지라 이슈 생성
  - 뱅크샐러드 사례: 슬랙 채팅으로 배포 명령(`deploy experiment production <commit-id>`), 서버 최신 배포 상태 조회(`status <repo-name>`) 가능

---

## 3. 왜 슬랙봇을 만들어야 하는가

- 전 사원이 **압도적으로 가장 오래 사용하는 툴 = 메신저(슬랙)**
- 업무 중 항상 켜져 있는 하나의 허브에서 알림 수신 + 외부 도구 조작 가능
- 개발·비개발 모두 공통으로 사용하는 툴에서 업무 흐름을 제어 → **전사 생산성 향상**

## 4. 슬랙앱 만들기

### 앱 생성 (api.slack.com)

1. `Your Apps` → `Create New App`
2. **From scratch** 선택 (From App Manifest는 JSON 직접 작성 필요 → 어려움)
3. 앱 이름·워크스페이스 설정 → 앱이 슬랙 클라우드에 업로드됨 (아직 설치 아님)

### 앱 설정 핵심 두 가지

- **Features & Functionality** 설정
  - 다양한 기능 메뉴 존재(Incoming Webhooks, Slash Commands 등)
  - 최초 필수 설정: **Bots** → 봇 유저 생성
    - `App Display Name` 설정 → 슬랙 내 봇 유저 한 명 생성됨 (예: Jira 봇, Donut 봇)
- **Install your app** → 워크스페이스에 설치
  - 설치 후 **User OAuth Token** + **Bot User OAuth Token** 발급
  - 토큰 발급 완료 = 앱이 슬랙 이벤트를 수신할 준비 완료

## 5. 봇 서버 만들기

### 개발 방식 세 가지

#### (1) Incoming Webhook

- 앱 설정에서 Webhook URL 생성 → 해당 URL 호출 시 슬랙 메시지 전송
- **장점**: 코드 거의 없이 구현 가능, 외부 서비스에서 URL만 등록하면 알림 발송
- **단점**: 슬랙 메시지 전송 외 다른 동작 불가
- 용도: 알림 전용으로 충분할 때

#### (2) Bolt 프레임워크

- 슬랙 소통에 최적화된 **서버 프레임워크**
- 지원 언어: JavaScript/TypeScript, Python, Java
- 예시 코드 (JavaScript):
  ```js
  const app = new App({ token: process.env.SLACK_BOT_TOKEN });
  // 이벤트 리스너 등록
  app.message('hello', async ({ message, say }) => { ... });
  // 메시지 전송
  await say('안녕하세요!');
  // 서버 시작
  await app.start();
  ```
- 봇 토큰을 `App` 객체에 연결 → 서버와 슬랙 앱 연결
- 슬랙 내 다양한 사용자 동작(채팅, 채널 입퇴장, 아카이브 등) 모두 리스닝 가능
- 레퍼런스: Bolt 공식 홈페이지 (Bolt-JS, Bolt-Python, Bolt-Java)

#### (3) Slack SDK

- 슬랙 소통 기능을 미리 갖춘 **패키지/라이브러리** (프레임워크보다 작은 단위)
- 지원 언어: Node.js, Python, Java
- **서버 프레임워크는 별도 선택** (Spring, Django, Flask, NestJS 등 자유)
- 핵심 패키지: **Web API** (130개 이상 메소드)
- 예시 코드 (NestJS 기반):
  ```ts
  const client = new WebClient(process.env.SLACK_BOT_TOKEN);
  // 메시지 전송
  await client.chat.postMessage({ channel, text });
  ```
- 발표자 선택: **SDK + NestJS** — 서버 환경 커스터마이징 자유도가 높아서

### Bolt vs SDK 비교

| | Bolt | SDK |
|---|---|---|  
| 서버 프레임워크 | 내장 | 직접 선택 |
| 슬랙 소통 메소드 | 내장 | 별도 패키지 |
| 자유도 | 낮음 | 높음 |
| 구현 편의성 | 높음 | 상대적으로 낮음 |

### 외부 서비스 연동

- 슬랙 이벤트를 받는 컨트롤러(NestJS)에서 외부 서비스 API 호출
- 연동 방식: REST API, gRPC 등 자유 선택
- 슬랙 구성도 요약:
  - 슬랙 앱 ↔ (Bot Token) ↔ 서버 ↔ 외부 서비스

## 6. 설정 더보기

### 토큰 권한(Scopes) 설정

- 메소드 사용 전 공식 문서에서 **Required scopes** 확인
- 예: `chat.postMessage` 사용 시 `chat:write` 스코프 필요
- 콘솔 → OAuth & Permissions → Scopes에서 봇 토큰 권한 추가 후 재설치

### 이벤트 수신 설정 (Event Subscriptions)

- 콘솔 → Event Subscriptions → 스위치 ON
- 수신할 이벤트 구독 추가 (예: `app_mention`, `message.channels`, `channel_deleted`)
- **Request URL**: 배포된 서버 도메인 입력 → 이벤트 발생 시 해당 URL 호출

### 소켓 모드 (Socket Mode)

- 문제 상황: 기업 엔터프라이즈 환경에서 서버가 내부망 → 슬랙이 서버로 직접 호출 불가
- 해결책: **소켓 모드** — 앱과 서버를 직접 연결 (URL 노출 불필요)
- 설정:
  1. 콘솔 → Socket Mode → 스위치 ON
  2. **App-Level Token** 별도 생성 (기존 Bot Token과 별개)
  3. `SocketModeClient` 사용 (기존 `WebClient`와 다른 클라이언트)
  ```js
  const socketClient = new SocketModeClient({ appToken: process.env.SLACK_APP_TOKEN });
  socketClient.on('app_mention', async ({ event }) => { ... });
  ```
- 이벤트 이름은 공식 레퍼런스에서 확인

> 레퍼런스를 자주 봐야 하는 세 가지: 받고 싶은 이벤트 이름, 쓰고 싶은 메소드 이름, 필요한 권한(Scope)

## 7. 슬랙봇 활용 예시

### 예시 1 — 휴가 인원 자동 알림

- 매일 오전 9시 스케줄러 → 인사 DB 조회 → 슬랙봇 서버 호출 → 특정 채널에 오늘 휴가 인원 알림
- 효과: 별도 검색·문의 없이 매일 자동 수신

### 예시 2 — 항공 예약률 조회 (에어프레미아 특화)

- 직원이 슬랙 채팅으로 명령 입력: "5월 7일 NRT 예약률 알려줘"
- 슬랙봇 서버 → 예약 DB/Tableau 조회 → 결과 슬랙으로 회신: "5월 7일 NRT 50석 남음"
- 효과: DB·Tableau 접근 권한이 없는 비개발 직원(승무원, 정비사 등)도 사용 가능 → **접근성 민주화**

### 예시 3 — 비행 채널 자동 생성·아카이브

1. 스케줄러가 내일 비행 스케줄 조회 → 슬랙봇 서버 호출
2. 해당 비행 채널 자동 생성 + 관련 조종사·승무원 자동 초대
3. 비행 중 커뮤니케이션 진행
4. 비행 1~2일 후 채널 자동 아카이브 + 평가 링크 발송
- 효과: 채널 생성·인원 초대·아카이브·히스토리 관리 전 과정 자동화

## 8. 마무리

- **슬랙봇 영향력**: 개발팀뿐 아니라 전 사원이 활용 → **전사 생산성 향상**
- 무엇을 만들지 막막할 때: "우리 업무 중 자동화할 수 있는 일이 무엇인가?"에서 출발
- 업무 자동화는 혼자 하는 것이 아니므로 **기획 단계부터 다양한 의견 수렴** 필요
- 발표자 강조: 이런 경험(전사 생산성 향상에 직접 기여, DB 접근 권한, 커리어 성과)은 스타트업 초창기가 아니면 얻기 어려운 기회
