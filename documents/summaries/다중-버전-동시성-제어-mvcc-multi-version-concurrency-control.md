## 동시성 문제와 전통적 해결책 — 락(Lock)

- 수십~수천 명이 동시에 데이터베이스에 접속하면 **데이터 부정합** 문제가 발생한다.
- 여러 사용자가 동일 데이터를 동시에 수정하면 데이터가 누더기가 되는 현상.
- 기존 해결책: **락(Lock)** — 한 사용자가 데이터를 수정하는 동안 다른 사용자의 접근을 차단.
  - 장점: 데이터 정합성 보장.
  - 단점: 대기 줄 발생, 전체 시스템 처리 속도 저하.

## MVCC 개요

- **MVCC(Multi Version Concurrency Control, 다중 버전 동시성 제어)**: 락 대신 데이터의 여러 버전을 관리하여 동시성을 확보하는 기법.
- 핵심 아이디어: 읽는 사람과 쓰는 사람이 서로 방해하지 않고 각자 작업할 수 있도록 함.

## 동작 원리 — 타임스탬프 기반 버전 관리

- 모든 **트랜잭션**은 시작 시 고유한 **타임스탬프**를 부여받는다.
- 데이터베이스 내 각 데이터도 버전별로 타임스탬프를 가진다.
- 트랜잭션이 데이터를 읽을 때, 시스템은 트랜잭션의 타임스탬프와 데이터의 버전 타임스탬프를 비교하여 해당 시점에 맞는 버전을 제공한다.
- 결과적으로 각 트랜잭션은 자신이 시작된 순간의 **스냅샷**을 독립적으로 조회하게 되어, 다른 사용자의 변경에 영향받지 않는다.
- 이를 통해 마치 모든 작업이 순서대로 직렬 실행되는 것처럼 보이는 **직렬 가능성(Serializability)**이 보장된다.

> (강의 비유) "MVCC는 똑똑한 도서관 사서 같은 것. 작가가 원본을 수정하는 동안, 독자들에게는 수정 전 복사본을 나눠줘 서로 방해받지 않게 한다."

## 내부 동작 단계 — 업데이트 시 흐름

1. **업데이트 요청** 수신.
2. 기존 데이터를 **언두(Undo) 영역**(임시 대피소)에 백업.
3. 원래 공간을 **새로운 데이터**로 업데이트.
4. 사용자가 **커밋(Commit)** 하면 **SCN(System Change Number)**이 1 증가.
- 이 과정 동안 다른 사용자들은 언두 영역의 이전 버전 데이터를 간섭 없이 계속 읽을 수 있다.

## 핵심 구성 요소

- **SCN(System Change Number)**: 시스템 전체의 시계 역할. 트랜잭션이 커밋될 때마다 증가.
- **언두(Undo) 영역**: 데이터가 변경될 때마다 이전 버전을 보관하는 저장소 (도서관 비유의 '서고').
- **CR카피(Consistent Read Copy)**: 일관된 읽기(Consistent Read)를 위한 복사본. SCN을 기준으로 쿼리가 요청한 시점의 데이터 버전을 언두 영역에서 찾아 제공.

## 트레이드오프 — MVCC의 비용

- 높은 동시성을 얻는 대신 시스템 자원을 추가로 소비하는 구조.

| 비용 유형 | 설명 |
|---|---|
| **I/O 오버헤드** | 오래된 데이터를 읽기 위해 언두 영역까지 디스크 접근이 증가 |
| **스토리지 오버헤드** | 데이터의 이전 버전들을 계속 보관해야 하므로 저장 공간 추가 소요 |
| **CPU 오버헤드** | 버전 관리 및 버전 선택 계산으로 CPU 부하 증가 |

## 실무적 시사점

- **읽기 동시성**과 **쓰기 처리량**은 매우 높고, **자원 비용**은 중간~높음 수준.
- 서비스 성능 저하 시 단순히 "느리다"고 느끼는 것을 넘어, "이 쿼리가 오래된 데이터를 찾느라 언두 블록을 계속 뒤지며 I/O가 과부하되는 것은 아닌가"라는 구체적 진단이 가능해짐.
- 현대 데이터베이스(Oracle, PostgreSQL 등)는 대부분 MVCC를 채택하고 있으며, 내부 버전 관리가 시스템 성능에 미치는 영향을 이해하는 것이 중요하다.
