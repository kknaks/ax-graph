## 배경: 데이터 폭증의 위기

- **IO 병목 심화**: 데이터 증가 → 인덱스 깊이 증가 → 디스크 접근 비용 상승
- **동시성 충돌 증가**: 대용량 트랜잭션 경합·락 대기 시간 증가
- **운영 비용 폭증**: 백업·복구·배포 시간 급증
- **물리적 서버 한계 도달**: CPU·메모리·디스크 단순 확장만으로는 한계

→ 해결책: 데이터를 **논리적** 또는 **물리적**으로 분리 — **파티셔닝** 또는 **샤딩** 적용

---

## 파티셔닝 (Partitioning)

> "한 지붕 아래 여러 개의 방을 만드는 개념"

- **정의**: 단일 DB 서버 내부에서 하나의 거대한 테이블을 여러 **파티션(저장 단위)**으로 분리하는 전략
- 개발자는 하나의 테이블처럼 사용 가능 → DB 내부에서 여러 파티션으로 분리
- 파티션 접근·라우팅은 **DB 엔진(옵티마이저)**이 자동 처리
- 예시: 단일 DB 인스턴스 안에서 연도별로 파티션 분할

### 파티셔닝 종류

1. **레인지(Range) 파티션** — 날짜·숫자 범위 기준 분할
   - 예: 월별·연도별 로그 데이터 (게임 회사 로그 테이블 실사례)
2. **리스트(List) 파티션** — 특정 값의 목록 기준 분할
   - 예: 지역 코드(서울·부산 등)
3. **해시(Hash) 파티션** — 특정 컬럼에 해시 함수 적용 → 균등 분산
   - 목적: 특정 값 쏠림 방지
4. **컴퍼지트(Composite) 복합 전략** — 두 가지 이상 전략 조합
   - 예: 범위 + 해시 혼합

> 비즈니스 도메인 특성·데이터 규모·데이터 특성에 따라 적절한 전략 선택

## 샤딩 (Sharding)

> "옆 동네에 새 집 짓기" — 파티셔닝이 '한 지붕 아래 새 방'이라면, 샤딩은 '아예 새로운 집으로 이사'

- **정의**: 데이터를 물리적으로 다른 DB 인스턴스로 분리·저장하는 전략
- 각 서버를 **샤드(Shard)**라고 부름
- **스케일 아웃(수평 확장)**의 핵심 기술
- 서로 다른 물리적 DB 인스턴스에 샤드 키 기반으로 데이터를 분산 저장
- **샤드 키 기반 라우팅 로직**을 애플리케이션 또는 미들 레이어에서 직접 구현해야 함

---

## 파티셔닝 vs 샤딩 핵심 비교

| 항목 | 파티셔닝 | 샤딩 |
|---|---|---|
| **물리적 위치** | 단일 DB 인스턴스 내부 | 다수의 별도 DB 인스턴스 |
| **확장 방향** | 스케일업(단일 서버 최적화) | 스케일아웃(수평 확장) |
| **관리 주체** | DB 엔진·옵티마이저 자동 처리 | 애플리케이션·미들 레이어에서 구현 |
| **SQL 변경** | 불필요 (개발자는 파티션 존재 몰라도 됨) | 샤드 키 설계 필수 |
| **아키텍처 레벨** | DB 내부 최적화 기술 | 아키텍처 레벨 분산 전략 |

## 실무 활용

### 파티셔닝 적합 사례
- **로그 테이블**: 시간 기준으로 지속 누적되는 대용량 테이블
- **날짜 기반 최신 데이터 조회 성능**이 중요할 때 (조회 조건에 날짜 컬럼 자주 사용)
- **대량 오래된 데이터 일괄 정리**: 파티션 단위로 삭제 → 효율적

### 샤딩 적합 사례
- **대규모 트래픽 + 대용량 데이터**: 단일 DB 한계를 넘을 때
- **글로벌 서비스**: 지역별(한국·미국 등) 데이터 물리 분산 → 데이터 로컬리티 확보
- **스케일업 한계 도달**: 단일 서버 사양을 아무리 높여도 성능 개선 불가할 때

### 샤딩 주의사항
- **조인 연산이 거의 불가능** → 사전에 데이터 모델링을 철저히 설계해야 함
- 샤딩은 **최후의 수단**에 가까움 — 단일 DB 한계를 넘었을 때 도입 검토

---

## 정리

- **파티셔닝**: 단일 서버 내 조회 성능·관리 효율을 위한 테이블 분할 → **내부 최적화 전략**, SQL 변경 없이 투명하게 적용
- **샤딩**: 하드웨어 한계를 넘기 위해 데이터를 여러 물리 서버로 분산 → **물리적 확장 전략**, 애플리케이션 레벨 분산 로직 필요·조인 제약 발생
