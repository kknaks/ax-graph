## 문제 제기: 코더와 아키텍트를 가르는 기준

- AI가 코드를 써 주는 시대에도 진짜 개발자라면 알아야 하는 숫자가 있다.
- 회의 시나리오: 주니어가 "DB 응답이 느려서 Redis 캐시를 도입하고 싶다"고 하자 시니어가 "얼마나 느린데, 캐시로 얼마나 빨라지냐"고 물었고 중간 개발자는 답을 못함.
- **코더와 아키텍트의 차이**: 코드를 잘 쓰는가가 아니라 **근거 있는 의사 결정**을 할 수 있는가.
- 코드 작성은 이미 AI가 잘 한다. 아키텍트는 근거·기준·의사 결정 능력이 필요하다.

## Latency Numbers Every Programmer Should Know

### 개요

- 컴퓨팅 시스템(CPU, 메모리, 디스크, 네트워크)의 **시간 비용을 정량화**한 표.
- 백엔드·시스템 설계 분야에서 전 세계적으로 널리 인용되는 레퍼런스.
- 인용 자료: DDIA(백엔드 바이블), 알렉시의 시스템 디자인 면접 책(100만 부 이상 판매), ACM Queue, 구글 리서치·코넬 대학교 학술 자료.

### 이 표가 답해 주는 질문들

- 캐시를 도입할 가치가 있을까?
- 네트워크 호출 횟수를 줄이면 얼마나 빨라질까?
- 데이터를 메모리에 두는 것이 정말 큰 차이가 날까?
- 리전을 분리하면 응답이 얼마나 느려질까?
- SSD가 RAM의 대안이 될 수 있을까?

## 수치의 역사: 세 인물

### 1. 피터 노빅 (Peter Norvig, 2001)

- **구글 리서치 디렉터** 출신, AI 교과서 《Artificial Intelligence: A Modern Approach》 공저자.
- 2001년 에세이 "Teach Yourself Programming in Ten Years"에서 24시간·21일 독학류 책들을 비판하며 "진짜 프로그래밍은 10년이 걸린다"고 주장.
- 이 에세이에 **11가지 레이턴시 수치표**가 최초로 포함됨.

**노빅의 주요 수치 (2001년 기준)**

| 연산 | 레이턴시 |
|---|---|
| 전형적인 CPU 명령 실행 | 1 ns |
| L1 캐시 참조 | 0.5 ns |
| L2 캐시 참조 | 7 ns |
| 메인 메모리 참조 | 100 ns |
| 1 Gbps 네트워크로 1 KB 전송 | 20,000 ns (20 μs) |
| 메모리에서 1 MB 순차 읽기 | 250 μs |
| 디스크 탐색(HDD) | 8 ms |
| 디스크에서 1 MB 순차 읽기 | 20 ms |
| 미국→유럽 패킷 왕복 | 150 ms |

- **노빅의 진짜 강조점**: 정확한 수치 암기가 아니라 **차수의 감각** — CPU(ns) vs 메모리(100 ns) vs 디스크(ms) vs 네트워크(ms) 간의 격차 크기를 아는 것.
- 관점: **단일 PC** 기준, 프로그래머가 알고리즘을 짤 때 마주하는 로컬 하드웨어의 물리적 제약.

### 2. 제프 딘 (Jeff Dean, 2009)

- **MapReduce, BigTable, TensorFlow** 공동 설계자, 구글의 살아 있는 전설.
- 2009년 LADIS(대규모 분산 시스템 학회) 키노트에서 노빅의 표를 구글 분산 시스템 경험 기반으로 확장.

> "중요한 것은 실제로 만들지 않고도 시스템의 성능을 추정할 수 있는 능력이다." — 제프 딘

**노빅 표 대비 추가된 수치**

| 연산 | 레이턴시 |
|---|---|
| 1 KB Gzip 압축 | 3 μs |
| SSD 랜덤 읽기 (4 KB) | (수치 추가) |
| SSD 순차 읽기 (1 MB) | (수치 추가) |
| 데이터센터 내부 RTT | 500 μs |
| 캘리포니아 → 네덜란드 패킷 왕복 | ~150 ms (노빅과 동일) |

- **압축 트레이드오프 사례**: 네트워크 직접 전송 10 μs vs 압축(3 μs) 후 전송(5 μs) = 8 μs → 압축이 2 μs 이득. "압축이 전송보다 빠를 때만 의미 있다"는 정량적 판단 근거.
- 관점: **분산 컴퓨팅** 시대, 데이터센터 내부 통신과 SSD라는 새로운 계층 추가.
- 참고: 아카(Akka) 창시자 **조나스 보너(Jonas Bonér)**가 2012년 제프 딘의 2009년 강연을 GitHub Gist로 정리하며 SSD 항목 2개를 추가한 버전이 현재 업계 표준으로 인용됨.

### 3. 콜린 스콧 (Colin Scott, 2012~)

- UC Berkeley 출신.
- 하드웨어는 시간에 따라 빠르게 진화하므로 수치를 **시간의 함수로 모델링**.
- 연도 슬라이더로 해당 시점의 레이턴시를 자동 계산하는 **인터랙티브 웹페이지** 제작.
- **진짜 가치**: 1990년이든 2020년이든 CPU 캐시 → 메모리 → SSD → HDD → 네트워크라는 **레이턴시 서열과 격차는 변하지 않음**을 시각적으로 체감.
- **한계**: 메모리·네트워크처럼 개선 속도가 제한적인 영역에서는 실제 수치와 차이 날 수 있음. 정확한 수치가 필요하면 최신 벤치마크를 참조해야 함.

## 실무 활용: 의사 결정 사례 3가지

### 시나리오 1: 캐시 도입 의사 결정

- 상황: HDD 기반 레거시 DB 조회가 느려 Redis 캐시 도입 검토.
- 수치 기반 판단:
  - HDD 디스크 탐색: ~10 ms
  - 데이터센터 내부 RTT: 500 μs
  - Redis 메모리 조회: 수십 ns
- 결론: HDD 대비 Redis가 차수가 다를 정도로 빠르므로 도입 가치 있음.
- 단, SSD 기반 DB라면 성능 차이 줄어들어 **부하 분산 관점**으로 재검토 필요.

### 시나리오 2: 마이크로서비스 분리의 진짜 비용

- 상황: 특정 기능을 별도 MSA로 분리할지 결정.
- 수치 기반 판단:
  - 로컬 메모리 내 함수 호출: 수 ns
  - 데이터센터 내부 네트워크 호출: 500 μs
  - 차이: **10만 배 이상** (+ 직렬화·역직렬화 비용 추가)
- 결론: 호출 빈도가 높다면 신중하게 접근, 무조건 분리가 답은 아님.
- 단, MSA는 레이턴시 외에 **확장성·독립성** 등 아키텍처 관점도 함께 고려해야 함.

### 시나리오 3: 시스템 디자인 면접 (100만 QPS)

- 질문: "URL 단축 서비스를 100만 QPS로 처리하려면?"
- 수치 기반 추론:
  1. 메모리 조회 100 ns → 이론상 단일 서버 초당 1,000만 건 처리 가능 → 메모리에서 끝나는 작업이면 한 대로 충분.
  2. 디스크 DB 조회 수 ms → 단일 서버로 100만 QPS 불가, 초당 수백 건도 버거움.
  3. 네트워크 호출 500 μs → 서비스 간 호출 횟수 줄이지 않으면 병목.
- 이 감각으로 캐시 필수, 샤딩 필요, 호출 최소화 이유를 **숫자로 정량화**해 설명 가능.

## 수치 활용 시 주의점 4가지

1. **정확한 수치를 외우려 하지 마라** — 실제 환경과 정확히 일치하지 않는 근사값. 핵심은 ns·μs·ms 단위 차이의 감각.
2. **전부 외우지 않아도 된다** — L1 캐시, 메인 메모리, SSD, 데이터센터 내부 RTT, HDD, 대륙 간 패킷 전송 정도만 알아도 의사 결정 가능.
3. **절대값이 아닌 상대적 격차(배수)가 중요하다** — 연도마다 절댓값은 변하지만 "RAM이 SSD보다 약 200배 빠르다" 같은 비율 감각은 유지됨.
4. **레이턴시만이 전부가 아니다** — 실무에서는 **처리량(Throughput)**도 함께 고려. 배치·병렬 처리로 레이턴시 비용을 상쇄할 수 있는 환경도 있음.

## 결론: 바이브코딩 시대의 본질

- 구조 설계·하중 계산을 모르는 건축가, 재료와 불의 성질을 모르는 요리사, 악보도 못 읽는 작곡가는 없다.
- 개발 본질에 관심 없이 AI에만 의존해 쏟아낸 코드는 결국 저품질 시스템이 된다.
- 바이브코딩 시대일수록 **아키텍처적 감각이 진짜 경쟁력**.
- 저자의 모토: "기본에 충실하고 중요한 것에 집중하자."
