## DB 커넥션 풀 개요

- **커넥션 풀**: DB 커넥션을 미리 생성해 두고 필요 시 빌려쓰고 반납하는 패턴
- 커넥션 생성 비용 항목
  - TCP 핸드셰이크
  - SSL/TLS 협상(보안 사용 시)
  - DB 계정 인증
  - 백엔드 프로세스 할당 (PostgreSQL은 커넥션당 별도 프로세스 → 메모리 증가)
- 생성 비용: 보통 수 ms ~ 수십 ms → 매 요청마다 생성하면 오버헤드 누적
- 풀 미사용 시: 매 쿼리마다 생성→소멸 반복 / 풀 사용 시: 초기 1회 생성 후 재사용

## 언어별 기본 풀 사이즈

| 언어/라이브러리 | 기본 최대 풀 사이즈 |
|---|---|
| Java (HikariCP, Spring Boot 2.0+) | **10** |
| .NET (ADO.NET 내장 풀) | **100** |
| Node.js (node-postgres pg pool) | **10** |
| Python (SQLAlchemy QueuePool) | **5** (overflow 10, 최대 15) |

- .NET 100개는 MS SQL Server의 맥스 커넥션(~32,000)이 충분히 크기 때문; Microsoft 공식 가이드는 "기본값 그대로 쓰지 말고 워크로드에 맞게 조정하라"고 명시
- .NET을 제외한 나머지 라이브러리들이 5~15 범위에 수렴 → 기술적 근거가 있음을 시사

## 풀 사이즈 설정 실무 패턴 (일반적 관행)

- 개발·스테이징: HikariCP 기본값 10 그대로 사용
- 소규모 서비스: 10
- 중소규모: 10~30
- 트래픽 많은 서비스: 30~50 (부하 테스트 검증 후)
- **100개 이상은 안티패턴** — 너무 많음

## 핵심 역설: 커넥션 수가 많을수록 느려진다

- **일반적 오해**: 일꾼이 많으면 일이 빨리 끝난다 → DB에서는 반대
- Oracle RWP(Real-World Performance) 문서 인용:
  > "연결 수를 줄이면 CPU 부하가 줄어들어 응답 속도가 빨라지고 처리량이 증가한다"
- **HikariCP 위키 벤치마크**: PostgreSQL에서 약 50 커넥션 부근부터 TPS가 평탄화됨
- **Oracle 권장**: CPU 코어당 평균 프로세스 수 10개 이하 유지; 코어 수 기준(스레드 수 기준 아님)

### 느려지는 이유 (USL — Universal Scalability Law)

1. **컨텍스트 스위칭 비용**: 코어 8개에 스레드 200개면 CPU가 스레드 교체에 시간을 소모; 스레드 교체 시 L1/L2 캐시도 무효화
2. **락 경쟁 증가**: 같은 테이블·행을 노리는 스레드가 많아질수록 락 대기 시간이 처리 시간을 역전
3. **캐시 효율 저하**: 많은 커넥션이 각자 다른 데이터를 읽으면 DB 버퍼 캐시가 빠르게 교체 → 캐시 미스 증가 → 디스크 I/O 증가

> 핵심: "동시에 일하는 커넥션은 CPU가 처리할 수 있는 양만큼만 의미가 있다" — 스레드 풀, HTTP 클라이언트 풀 등 모든 풀에 동일 원리 적용

## 풀 사이즈 산정 공식

### 주 공식 (PostgreSQL 위키 + HikariCP 위키)

```
최대 풀 사이즈 = (물리 CPU 코어 수 × 2) + 유효 스핀들 수
```

- 출처: PostgreSQL 위키, 수년간 다양한 벤치마크에서 경험적으로 수렴된 값
- HikariCP 공식 가이드: "다른 DB에도 대체로 적용 가능"
- **엄밀히는 활성 커넥션 수에 대한 공식**이나, 실무에서는 최대 풀 사이즈 가이드로 활용

#### 코어 × 2인 이유
- DB 쿼리는 **CPU 작업 ↔ I/O 대기** 반복
- I/O 대기 동안 코어가 유휴 상태 → 다른 커넥션 1개를 끼워 넣으면 CPU 활용률 ~100%
- × 3, × 4는 컨텍스트 스위칭·캐시 무효화·락 경쟁 비용 발생으로 오히려 역효과

#### 하이퍼스레딩 제외 이유
- 논리 코어는 진짜 연산 자원이 아님
- DB 작업처럼 메모리 접근 빈번·캐시 미스 많은 작업에서 HT 효과 미미
- 2개의 HT 스레드가 같은 캐시를 두고 경쟁 → 성능 저하 가능
- 예: 8코어 16스레드 CPU → 공식에는 8 대입

#### 스핀들 수
- HDD 시대 개념: 물리 디스크 회전축 수 (RAID 4개 = 스핀들 4)
- **2026년 현재 NVMe SSD 환경: 스핀들 = 0** (회전축 없음, I/O 대기 거의 없음)
- 데이터가 메모리에 캐시된 경우도 스핀들 = 0
- **실질 공식: 풀 사이즈 = 물리 코어 수 × 2**

### 보조 공식 — 데드락 회피 최소 커넥션 수 (HikariCP 문서)

```
최소 커넥션 수 = Tn × (Cm - 1) + 1
(Tn: 동시 스레드 수, Cm: 한 스레드가 동시에 점유하는 최대 커넥션 수)
```

- 중첩 트랜잭션 등 Cm > 1인 코드에서 데드락 방지용
- 일반적인 앱은 Cm = 1 → 공식 결과 = 1 (의미 없음)
- **풀 사이즈 설정 공식이 아니라 데드락 발생 시 진단 도구**로 활용

## 리틀의 법칙으로 진단하기

```
TPS = 활성 커넥션 수 ÷ 쿼리 실행 시간(초)
```

### 예시 시나리오
- 풀 사이즈: 16, 평균 활성 커넥션: 14, 쿼리 실행 시간: 50ms
- 이론 TPS: 14 ÷ 0.05 = 280
- 활성 커넥션이 맥스의 ~90% 도달 → 처리량 증가 방법 검토 필요

### 처방 순서
1. **DB 코어 × 2 한계에 도달한 경우**: 풀 확장보다 쿼리 응답 시간 단축 우선
   - 쿼리 실행 시간 절반 → TPS 2배
2. **DB 서버 코어 여력 있는 경우**: 맥스 풀 사이즈 확장 검토
3. 위 두 방법으로 목표 TPS 미달 시: DB 자원 증설(코어 추가, 레플리카 활용)

> 리틀의 법칙은 "풀 크기를 정해 주는 공식"이 아니라 **"풀 부족 vs 쿼리 부족을 구별하는 진단 도구"**

## 단일 WAS 시나리오 예시

- 환경: DB 8코어, SSD, 충분한 RAM, 단일 WAS
- P95 쿼리 시간: 50ms, 목표 TPS: 300
- 시작 풀 사이즈: 8 × 2 + 0 = **16**
- 이론 상한 TPS: 16 ÷ 0.05 = 320; 실효 TPS(80%): **256** → 목표 300 미달
- 단계적 접근:
  1. 쿼리·인덱스 최적화로 P95 50ms 단축
  2. 풀 사이즈 18~20으로 점진 확장
  3. 그래도 미달 시 DB 자원 증설

## 다중 WAS 환경에서의 분배

- **흔한 실수**: WAS별로 독립적으로 풀 사이즈 16 지정 → WAS 4대 × 16 = 총 64개 → DB 허용치 초과
- **올바른 접근**: DB 전체 허용 커넥션을 WAS 수로 분배

```
각 WAS 풀 사이즈 = (DB 코어 × 2) ÷ WAS 인스턴스 수
```

예: 8코어 DB, WAS 4대 → 각 WAS = 16 ÷ 4 = **4개**

### DB 보호 vs 가용성 트레이드오프

| 우선순위 | WAS별 풀 사이즈 | 총 커넥션 | 특징 |
|---|---|---|---|
| DB 보호 우선 | 4 | 16 | 보수적, WAS 1대 장애 시 나머지 부담 증가 |
| 가용성 우선 | 5~6 | 20~24 | WAS 1대 장애를 견딜 여유 확보 |

- **오토스케일링 환경**: WAS 수가 동적으로 변하므로 **PgBouncer** 같은 외부 풀러 사용 권장

## DB 맥스 커넥션 제약

- PostgreSQL 기본 맥스 커넥션: **100** (시스템·관리자 예약 후 실사용 가능 ~90)
- **필수 조건**: 전체 WAS 풀 합산 ≤ DB 맥스 커넥션 × 0.8 (안전 마진)
- AWS RDS 가이드 예시: 맥스 500이면 풀러는 400으로 설정
- **장애 시나리오**:
  - WAS 5대 × 맥스 풀 30 = 150 > DB 맥스 100 → 초과
  - 고정 풀: 앱 시작 시 즉시 오류 (파드 기동 실패)
  - 동적 풀: 트래픽 폭증 시 100개 초과 지점에서 장애
- **맥스 커넥션을 단순히 올리는 것의 문제**: PostgreSQL은 프로세스 기반 → 커넥션당 수 MB ~ 수십 MB 메모리 소비, 컨텍스트 스위칭 누적
- **권장 대안**: 맥스 커넥션 증설 대신 **PgBouncer 같은 중간 풀러** 도입

## 기존 시스템 적용 시 주의사항

> "잘 동작하는 시스템은 건들지 말라" — 운영의 황금 룰

- 적용 전 **반드시 측정 먼저**: 피크 시간대 HikariCP 메트릭으로 평균 활성 커넥션 확인
- 판단 기준:
  - 활성 커넥션 / 맥스 풀 사이즈 < 25%: 풀 축소 안전
  - 25~65%: 신중하게 적용
  - 65% 이상: **풀 사이즈 줄이지 말 것**
- 단계적 적용 + 즉시 롤백 준비
- **이 가이드 적용처**: ① 트러블슈팅 진단 기준, ② 신규 시스템 구축 시 초기값

## 실무 함정 4가지

### 1. 롱 트랜잭션
- 트랜잭션 안에서 외부 API 호출, 무거운 계산 → 커넥션 점유 시간 증가 → 공식에 맞는 풀도 고갈 가능
- **처방**: 트랜잭션 최대한 짧게; 외부 호출은 트랜잭션 밖으로 이동
- **가장 흔한 풀 고갈 원인**

### 2. 유효 커넥션 타임아웃 미스매치
- **HikariCP maxLifetime** (기본 30분): 풀이 커넥션을 주기적으로 교체하는 주기
- **DB idle timeout**: DB가 유휴 세션을 강제 종료하는 시간
  - PostgreSQL: 0 (무제한)
  - MySQL: 8시간
- **문제**: maxLifetime > DB idle timeout → DB가 먼저 끊은 커넥션을 풀이 모르고 재사용 시도 → 오류
- **실무에서 자주 발생하는 경우**: 중간 계층 추가 시
  - PgBouncer server_idle_timeout 기본값 **10분** vs HikariCP maxLifetime 30분 → 즉시 미스매치
  - RDS Proxy, NLB 등 개입 시도 동일
- **처방**: maxLifetime을 DB 또는 중간 계층의 idle timeout보다 **30초 이상 짧게** 설정
- HikariCP GitHub README: "DB나 인프라가 부과하는 커넥션 시간 제한보다 몇 초 짧게 설정하라"

### 3. Cm > 1인 코드 (한 스레드가 커넥션 여러 개 동시 점유)
- 중첩 트랜잭션 패턴
- **처방**: 풀 크기 확장보다 코드 구조 수정 우선; 트랜잭션 분리 또는 JTA 활용

### 4. 커넥션 누수
- HikariCP `leakDetectionThreshold` 설정 (예: 60000 = 60초)
- 풀에서 나간 커넥션이 60초 이상 반환되지 않으면 로그 경고
- **핵심**: 풀 크기를 키우면 해결되겠지는 섣부른 처방; 대부분 고갈은 풀 크기가 아닌 커넥션 점유 시간 문제

## 핵심 요약 5가지

1. **커넥션 수가 많다고 좋지 않다** — 일정 수 초과 시 컨텍스트 스위칭·락 경쟁·캐시 교체로 처리량 감소
2. **물리 코어 × 2 (+스핀들)** — 풀 사이즈 시작점; 부하 테스트로 스위트스팟 탐색
3. **리틀의 법칙으로 검증** — 쿼리 실행 시간 단축을 TPS 증가의 첫 번째 수단으로
4. **작게 시작해서 측정하며 늘리기** — 임의의 큰 값으로 시작 금지
5. **롱 트랜잭션이 진짜 적** — 유발 요인 모두 제거

## 출근 후 즉시 점검 체크리스트 8가지

1. 맥스 풀 사이즈가 DB 물리 코어 × 2 기준인지 확인 (50·100 등 임의 큰 값 여부)
2. **고정 풀 운영**: HikariCP minimumIdle = maximumPoolSize (런타임 풀 변동 오버헤드 제거)
3. 커넥션 타임아웃 기본값 유지 (너무 길면 역효과)
4. `leakDetectionThreshold` 설정으로 커넥션 누수 로그 모니터링
5. 롱 트랜잭션 제거 (불필요하게 긴 로직은 트랜잭션 밖으로)
6. HikariCP 메트릭(active·pending·idle) 대시보드 구축
7. WAS 인스턴스 수 × 맥스 풀 합산 ≤ DB 맥스 커넥션 확인; 초과 시 PgBouncer 도입 검토
8. 부하 테스트로 새츄레이션 포인트(처리량이 꺾이는 지점) 파악
