## 디자인 패턴이란

- **설계(Design) + 패턴(Pattern)**: 여기서 '디자인'은 시각적 디자인이 아닌 **설계**를 의미함
- 특정 문맥에서 공통적으로 발생하는 문제에 대해 재사용 가능한 해결책(골격 구조)을 제공하는 개념
- 대부분의 개발자가 개발 과정에서 겪는 상황에 적용할 수 있는 정형화된 틀
- 완전한 정답 알고리즘이 아니라, 상황에 맞게 유연하게 수정해서 사용하는 것이 올바른 접근

## GoF(Gang of Four) 디자인 패턴

- 4명의 저자(GoF)가 제시한 **총 23개 패턴**이 가장 대표적으로 사용됨
- 현재 디자인 패턴은 수천 개가 존재하나, GoF 23개를 먼저 파악하는 것이 우선
- 생성·구조·행동 **3가지 범주**로 분류됨

## 디자인 패턴의 장점

- **개발자 간 원활한 소통**: 팀원 모두가 디자인 패턴을 인지하고 있으면 코드 구조 논의 및 이해가 수월해짐
- **소프트웨어 구조 파악 용이**: 패턴에 맞춰 작성된 코드는 처음 보는 사람도 이해하기 편함
- **재사용을 통한 개발 시간 단축**: 정형화된 클래스 구조를 그대로 복사하거나 약간 수정해 재활용 가능
- **설계 변경 유연성**: 중간 방향 전환이나 업데이트 시 비교적 원활하게 수정 가능

## 디자인 패턴의 단점

- **객체지향적 설계 고려 필요**: 소프트웨어 공학에서 제시된 패턴을 Java 등 객체지향 언어에 그대로 적용하면 객체지향 원칙에 맞지 않는 설계가 될 수 있음 → 언어 특성에 맞게 수정 반영 필요
- **초기 투자 비용 증가**: 패턴을 고려한 설계 후 착수하는 만큼 인력·시간·공수가 더 많이 소요됨

## GoF 23개 패턴 분류

### 생성 패턴 (Creational Patterns)
- **목적**: 객체 생성과 관련된 패턴. 특정 객체가 생성되거나 변경되어도 프로그램 구조에 최소 영향을 주는 유연성 제공
- **5가지 패턴**: 추상 팩토리(Abstract Factory), 빌더(Builder), 팩토리 메소드(Factory Method), 프로토타입(Prototype), **싱글톤(Singleton)**
- 외부 API 사용 시 이미 적용되어 있어 개발자가 자연스럽게 사용하게 되는 패턴들 포함

### 구조 패턴 (Structural Patterns)
- **목적**: 클래스/인터페이스/객체의 구조를 설계하는 패턴. 클래스나 객체를 조합해 더 큰 구조를 만들며, 기존 코드를 최소한으로 변경하면서 기능을 추가할 수 있게 해줌
- **강조 패턴**: **어댑터(Adapter)**, **브릿지(Bridge)**, **데코레이터(Decorator)**, **파사드(Facade)**

### 행동 패턴 (Behavioral Patterns)
- **목적**: 반복적으로 사용되는 객체들의 커뮤니케이션을 패턴화. 객체 사이의 알고리즘 또는 책임(Responsibility)을 분배하는 방법 정의
- **핵심 키워드**: **결합도(Coupling) 최소화** — 루즈 커플링(Loose Coupling) 지향
- 3가지 범주 중 패턴 수가 가장 많음
- **자주 언급되는 패턴**: State, 전략(Strategy), 인터프리터(Interpreter), 반복자(Iterator) 등
- 중요도는 프로젝트 상황에 따라 다르므로 일반적 우선순위를 단정하기 어려움

## 싱글톤(Singleton) — 스프링 부트와의 연관

- 강의에서 스프링 부트를 다루기 전 디자인 패턴을 소개하는 이유: **싱글톤 패턴을 필수적으로 알아야 하기 때문**
- **싱글톤 정의**: 클래스마다 인스턴스를 하나만 생성하여 어디서든 참조
- 주의: 소프트웨어 공학의 싱글톤과 스프링 부트에서 적용되는 싱글톤은 일부 차이가 있음
- 스프링 부트에서 완벽한 100% 재현은 어렵고, 스프링이 이미 제공하는 방식을 익혀 그 방식으로 코드를 작성하면 됨

## 실용적 활용 방법

- 모든 패턴을 암기할 필요 없음
- 어떤 패턴들이 존재하는지 간략히 파악 → 설계 단계에서 필요한 패턴을 검색 → 디테일을 더해 코드에 적용
- 이 방식으로 더 좋은 코드를 작성할 수 있음
